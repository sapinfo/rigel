## 1. Overview

### 1.1 Design Goals

1. **"렌더 로직은 1개"** — `FormRenderer.svelte` 가 화면과 PDF 모두의 유일한 source. `?print=1` 모드로 동일 컴포넌트 재사용.
2. **"코드 path 는 1개"** — `fn_advance_document` helper 가 순차(그룹 크기=1)와 병렬(그룹 크기>1) 를 한 로직으로 처리. v1.0/v1.1 회귀 zero.
3. **"해시는 JS 1개"** — 클라이언트(브라우저 crypto.subtle) 와 서버(Node crypto) 가 동일한 JCS + SHA-256 모듈 공유. DB 는 strict equality 저장만.
4. **"감사는 스냅샷"** — 서명 path, sha256, content_hash 는 상신·승인 시점에 audit_logs.payload 로 복사. 이후 profile 교체에 영향받지 않음.
5. **"drift 재발 금지"** — 0035 apply 직후 `pg_get_functiondef` 를 캡처하여 본 design §5 의 RPC signature 블록에 **자동** 반영하는 프로세스 도입 (§17).

### 1.2 Design Principles

- SvelteKit 2 + Svelte 5 runes + `@supabase/ssr` 스택 변경 금지
- v1.0/v1.1 RLS 규칙(`(SELECT auth.uid())`, FOR ALL 금지, action 별 분리) 완전 유지
- 모든 approval mutation 은 `SECURITY DEFINER` RPC 경유 (`ad_no_direct_insert` 패턴)
- Postgres enum `ALTER TYPE ADD VALUE` 는 v1.2 에서 사용하지 않음 (v1.1 M12 교훈 불필요)
- Forward-only migration — 0001~0030 수정 금지, 롤백은 문서화만

### 2.2 Data Flow — Submit (병렬결재 포함)

```
[Client: 기안 작성]
  1. FormRenderer 로 양식 내용 입력 (state)
  2. AttachmentInput → uploadAttachment.ts:
     a. Storage 업로드 (기존 경로 패턴)
     b. crypto.subtle.digest('SHA-256', fileBuffer) → sha256 hex
     c. approval_attachments INSERT { sha256 } (기존 RPC 유지, 신규 컬럼만 추가)
  3. ApprovalLine 편집:
     - "병렬 그룹 만들기" 버튼 클릭 → 선택된 연속 items 에 동일 groupOrder 부여
     - items[].groupOrder 할당 (0, 0, 1, 2, 2, ...)
  4. Submit 버튼 → <form use:enhance>
     use:enhance(({formData}) => {
       formData.set('approvalLine', JSON.stringify(lineWithGroupOrder))
       formData.set('attachmentIds', JSON.stringify(ids))
     })

[Server action: drafts/new/[formCode]/+page.server.ts]
  5. parse + zod validate
  6. import { computeDocumentHash } from '$lib/hash/documentHash'
     const contentHash = await computeDocumentHash({
       formSchema: form.schema,
       content: input.content,
       approvalLine: input.approvalLine,  // groupOrder 포함
       attachmentSha256s: attachments.map(a => a.sha256)
     })
  7. await supabase.rpc('fn_submit_draft', {
       p_tenant_id, p_form_id, p_content, p_approval_line,
       p_content_hash: contentHash,
       p_attachment_ids: ids,
       p_document_id: draftId ?? null
     })

[RPC 0035 fn_submit_draft v3]
  8. auth.uid() + is_tenant_member + content_hash 검증 (64-char hex)
  9. approval_line 검증:
     - jsonb array, length ≥ 1
     - 각 item: {userId, stepType, groupOrder(optional)}
     - groupOrder 미지정 시 step_index 로 fallback (순차)
     - 지정 시 set == {0, 1, ..., K-1} (연속, gap 금지)
     - 각 group 에 ≥ 1 approval step
     - 최소 1 approval step 전체
     - 모든 approver 가 tenant_members
  10. form FOR UPDATE → snapshot
  11. document INSERT (content_hash 저장, status='in_progress', current_step_index=0)
  12. 각 step INSERT with (group_order, step_index, status)
      v1.1 delegation 매칭 로직 재사용 — delegation 이 step 단위이므로 group 내 일부만 skipped 되어도 자연 작동
  13. PERFORM fn_advance_document(v_doc.id)
      — 첫 pending approval group 찾기 → current_step_index 갱신
      — 모든 approval step 이 delegation skipped → auto-completed
  14. attachments document_id UPDATE
  15. audit 'submit' INSERT (payload.content_hash mirror)
  16. RETURN v_doc
```

### 2.3 Data Flow — Approve (그룹 내 1건)

```
[Client: /documents/[id] → Approve 버튼]
Form action 'approve' (v1.1 유지, 수정 없음)
  → supabase.rpc('fn_approve_step', { p_document_id, p_comment })

[RPC 0036 fn_approve_step refactor]
  1. auth.uid() + doc FOR UPDATE
  2. 상태 체크: status IN ('in_progress', 'pending_post_facto') (v1.1 M13)
  3. is_tenant_member
  4. 현재 step 잠금 (step_index = current_step_index):
     — v1.1 순차 모델에서는 step_index 가 유일 활성 step
     — v1.2: current_step_index 의미 = "현재 pending group_order".
         group 내 여러 approver 가 있을 때, "내 차례" 는 RLS 로는 모든 멤버에게 열림.
         어느 step 이 "현재 사용자의 것" 인가? → step_index 가 아닌
         "group_order = current_step_index AND approver_user_id = auth.uid() AND status='pending'" 로 SELECT
  5. 위 단일 row 선택 (0 또는 1 row):
     - 0 row → proxy 분기 (v1.1 M12 유지):
         "group_order = current AND status='pending'" 중에서 absence.delegate 매칭되는 step 조회
         체인 깊이 1 강제 (proxy 자신도 absence 면 reject)
     - 1 row → 본인 직접 승인 (v1.0 경로)
  6. UPDATE step SET status='approved', acted_by_proxy_user_id
  7. audit INSERT 'approve' | 'approved_by_proxy'
     payload 추가: {
       signature_storage_path: <본인 또는 proxy 의 프로필 시그니처 path snapshot>,
       signature_sha256: <동일 snapshot>
     }
  8. PERFORM fn_advance_document(v_doc.id)
     — helper 내부에서 group 완료 체크 + 전진 or completed 판정
  9. RETURN v_doc
```

**중요 변경**: v1.1 의 `fn_approve_step` 은 `SELECT * FROM approval_steps WHERE step_index = current_step_index` 로 step 을 찾았음. v1.2 에서는 **group 에 여러 step 이 있을 수 있으므로 이 쿼리가 무의미**. `step_index` 대신 `group_order + approver` 조합으로 대체. 자세한 SQL 은 §5.4.

### 2.4 Data Flow — Reject / Withdraw

v1.0 로직 재사용. 병렬 그룹 내 1건 반려 → 전 pending step skipped + document rejected. fn_reject_step 은 `step_index` 필터를 `group_order = current_step_index AND approver_user_id = auth.uid() AND status='pending'` 로 교체 (fn_approve_step 과 동일 패턴).

### 2.5 Data Flow — PDF 요청

```
[Client]
  GET /t/{slug}/approval/documents/{id}/pdf

[Server: documents/[id]/pdf/+server.ts]
  1. locals.user 확인, resolveTenant, doc 권한 체크 (기존 documents/[id] load 와 동일 RLS)
  2. const cookie = event.request.headers.get('cookie') ?? ''
  3. const origin = event.url.origin
  4. const pdfBuffer = await renderDocumentPdf({ cookie, origin, docId: params.id })
  5. const { data: doc } = await locals.supabase
       .from('approval_documents')
       .select('doc_number')
       .eq('id', params.id).single()
  6. return new Response(pdfBuffer, {
       headers: {
         'Content-Type': 'application/pdf',
         'Content-Disposition': `inline; filename="${doc.doc_number}.pdf"`
       }
     })

[renderDocument.ts: singleton puppeteer]
  getBrowser():
    if (_browser === null) _browser = await puppeteer.launch({
      headless: 'new',
      args: ['--no-sandbox', '--font-render-hinting=none']
    })
    _browser.on('disconnected', () => { _browser = null })  // R11
    return _browser

  renderDocumentPdf({ cookie, origin, docId }):
    const browser = await getBrowser()
    const page = await browser.newPage()
    try {
      await page.setExtraHTTPHeaders({ cookie })
      await page.setViewport({ width: 794, height: 1123 })  // A4 at 96dpi
      const url = `${origin}/t/*/approval/documents/${docId}?print=1`
      // tenantSlug 는 origin 쪽에서 캐시한 값을 사용 (미리 resolve)
      await page.goto(url, { waitUntil: 'networkidle0', timeout: 30_000 })
      return await page.pdf({
        format: 'A4',
        printBackground: true,
        margin: { top: '20mm', right: '15mm', bottom: '20mm', left: '15mm' }
      })
    } finally {
      await page.close()
    }

  warmUp() (서버 boot 시 1회):
    await getBrowser()  // 싱글톤 Chromium launch

[Print mode: documents/[id]/+page.svelte]
  let { data, form } = $props()
  const print = $derived(new URLSearchParams(location.search).get('print') === '1')

    <DocumentHeader ... />
    <ApprovalPanel ... />
  {/if}
  <FormRenderer ... />
  <DocumentSteps ... />
  {#if !print}
    <AuditTimeline ... />
  {/if}
```

### 2.6 Data Flow — Signature Upload

```
[Client: /my/profile/signature]
  Form with <input type="file" accept="image/png">
  Client-side: file size ≤ 512KB guard + PNG magic bytes check

[Server action]
  1. resolveTenant
  2. File → ArrayBuffer
  3. Validate:
     a. bytes[0..3] === [0x89, 0x50, 0x4E, 0x47]
     b. size ≤ 524288
  4. sha256 = crypto.createHash('sha256').update(buf).digest('hex')
  5. Storage upload (upsert) → path `${user.id}/signature.png`
  6. supabase.from('profiles').update({
       signature_storage_path: path,
       signature_sha256: sha256
     }).eq('id', user.id)
  7. return { ok: true }
```

### 3.3 Database Schema — 신규/변경 컬럼 요약

| 테이블 | 컬럼 | 타입 | NULL | Default | CHECK |
|---|---|---|---|---|---|
| `approval_steps` | `group_order` | `int` | NOT NULL | - (backfill 후 DROP DEFAULT) | `group_order >= 0` |
| `approval_documents` | `content_hash` | `text` | NULL | - | `content_hash IS NULL OR char_length(content_hash) = 64` |
| `approval_attachments` | `sha256` | `text` | NULL | - | `sha256 IS NULL OR char_length(sha256) = 64` |
| `profiles` | `signature_storage_path` | `text` | NULL | - | - |
| `profiles` | `signature_sha256` | `text` | NULL | - | `signature_sha256 IS NULL OR char_length(signature_sha256) = 64` |

**의도적 NULL 허용**:
- `content_hash` — v1.0/v1.1 기존 문서는 backfill 불가 (원본 canonicalization 재현 시점 차이). "v1.2 부터의 신규 문서만 해시 증명 가능" 정책. CHECK 로 v1.2 이후 신규 상신은 NOT NULL 강제하고 싶지만 Postgres CHECK 는 "status != draft 인 경우 NOT NULL" 같은 조건을 일반 CHECK 로 표현할 수 있음. 단, 기존 v1.0/v1.1 문서가 이미 status != draft 이므로 CHECK 추가 시 위반. **결정**: CHECK 에 length 제약만 두고, NOT NULL 강제는 **RPC 내부 검증** 으로 이동 (`IF p_content_hash IS NULL OR char_length(p_content_hash) <> 64 THEN RAISE`).

- `sha256` — 기존 attachments backfill 불가. RPC 에서 v1.2 상신 path 에만 강제 검증 (attachment 의 `uploaded_at >= v1.2_cutoff` 조건 대신, 더 단순히 "submit 시 sha256 IS NULL 이면 fail" — 단 pre-existing draft 가 v1.1 시점 업로드 + v1.2 에서 상신하는 케이스는 fail 허용 가능 — 실무상 희박).

### 3.4 Indices

| 이름 | 테이블 | 컬럼 | 목적 |
|---|---|---|---|
| `as_document_group_idx` | approval_steps | `(document_id, group_order, status)` | fn_advance_document 쿼리 최적화 |
| `as_approver_group_idx` | approval_steps | `(approver_user_id, group_order) WHERE status = 'pending'` | inbox 쿼리 (v1.1 유지 + group_order 추가) |

기존 인덱스 `as_approver_pending_idx` 는 유지.

```sql
-- v1.2 M15: fn_advance_document helper 추출 (G4 이월).
--
-- v1.0/v1.1 의 fn_approve_step / fn_submit_draft 내부에 inline 되어 있던

--          0032_b 에서 fn_advance_document 를 group_order 버전으로 CREATE OR REPLACE.
--          (migration 분할 이유: 0032_a 는 ALTER/backfill, 0032_b 는 함수 교체)
-- 단순성을 위해 **0031 을 group_order 기준으로 작성**하고, group_order 컬럼은 동일 파일
-- 에서 먼저 ALTER + backfill + DROP DEFAULT 한 뒤 함수 생성. 1개 파일로 원자성 확보.

ALTER TABLE public.approval_steps
  ALTER COLUMN group_order DROP DEFAULT;

-- CHECK + 인덱스
ALTER TABLE public.approval_steps
  ADD CONSTRAINT as_group_order_nonneg CHECK (group_order >= 0);

CREATE INDEX IF NOT EXISTS as_document_group_idx
  ON public.approval_steps (document_id, group_order, status);

-- Step 2: fn_advance_document 함수 생성
CREATE OR REPLACE FUNCTION public.fn_advance_document(p_document_id uuid)
RETURNS public.approval_documents
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_doc public.approval_documents;
  v_current_group int;
  v_has_pending boolean;
  v_next_group int;
BEGIN
  -- 문서 잠금
  SELECT * INTO v_doc
  FROM public.approval_documents
  WHERE id = p_document_id
  FOR UPDATE;

  IF v_doc.id IS NULL THEN
    RAISE EXCEPTION 'Document not found: %', p_document_id USING ERRCODE = '02000';
  END IF;

  -- 종결 상태면 변경 없이 반환 (멱등)
  IF v_doc.status NOT IN ('in_progress', 'pending_post_facto') THEN
    RETURN v_doc;
  END IF;

  v_current_group := v_doc.current_step_index;

  -- 현재 group 내 pending approval step 이 있는가?
  SELECT EXISTS (
    SELECT 1
    FROM public.approval_steps
    WHERE document_id = p_document_id
      AND group_order = v_current_group
      AND step_type = 'approval'
      AND status = 'pending'
  ) INTO v_has_pending;

  IF v_has_pending THEN
    -- 그룹 내 아직 미완 approver 있음 → 전진 없음
    RETURN v_doc;
  END IF;

  -- 그룹 완료 → 다음 pending approval group 탐색
  SELECT MIN(group_order) INTO v_next_group
  FROM public.approval_steps
  WHERE document_id = p_document_id
    AND group_order > v_current_group
    AND step_type = 'approval'
    AND status = 'pending';

  IF v_next_group IS NULL THEN
    -- 모든 approval 완료 → document completed
    -- (v1.0 Risk #13: 남은 pending reference step 일괄 skipped)
    UPDATE public.approval_steps
    SET status = 'skipped'
    WHERE document_id = p_document_id
      AND status = 'pending';

    UPDATE public.approval_documents
    SET status = 'completed',
        completed_at = now(),
        updated_at = now()
    WHERE id = p_document_id
    RETURNING * INTO v_doc;
  ELSE
    -- 다음 그룹으로 전진
    UPDATE public.approval_documents
    SET current_step_index = v_next_group,
        updated_at = now()
    WHERE id = p_document_id
    RETURNING * INTO v_doc;
  END IF;

  RETURN v_doc;
END;
$$;

GRANT EXECUTE ON FUNCTION public.fn_advance_document(uuid) TO authenticated;

COMMENT ON FUNCTION public.fn_advance_document(uuid) IS
  'v1.2 M15 — group-based document advancement helper. Called from fn_submit_draft/post_facto and fn_approve_step. Handles parallel (group_order)와 sequential (group_order 증가)를 단일 로직으로 처리.';

COMMENT ON COLUMN public.approval_steps.group_order IS
  'v1.2 M15 — parallel group pointer. Same value = parallel approvers, increasing = sequential. Backfilled from step_index for pre-v1.2 rows.';

COMMENT ON COLUMN public.approval_documents.current_step_index IS
  'v1.0/v1.1: step_index pointer. v1.2 이후: group_order pointer (의미 재정의). 기존 문서는 group_order = step_index 이므로 값 동일.';

COMMIT;
```

### 4.5 `0035_fn_submit_v3.sql`

```sql
-- 0035_fn_submit_v3.sql
-- v1.2 M16: fn_submit_draft / fn_submit_post_facto 교체 — content_hash + groupOrder 지원.
-- 시그니처 변경 있음 — 기존 호출부 수정 필요.

BEGIN;

-- ─── fn_submit_draft v3 ────────────────────────────

CREATE OR REPLACE FUNCTION public.fn_submit_draft(
  p_tenant_id      uuid,
  p_form_id        uuid,
  p_content        jsonb,
  p_approval_line  jsonb,
  p_content_hash   text,            -- NEW
  p_attachment_ids uuid[] DEFAULT '{}'::uuid[],
  p_document_id    uuid   DEFAULT NULL
)
RETURNS public.approval_documents
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_user_id uuid := auth.uid();
  v_doc public.approval_documents;
  v_schema jsonb;
  v_doc_number text;
  v_item jsonb;
  v_index int := 0;
  v_group_order int;
  v_approval_count int := 0;
  v_user_ids uuid[] := '{}';
  v_uid uuid;
  v_amount numeric(14,2) := NULL;
  v_step_type text;
  v_step_status public.step_status;
  v_step_comment text;
  v_delegation_rule record;
  v_group_orders int[] := '{}';
  v_max_group int := -1;
  v_missing_attachments int;
BEGIN
  IF v_user_id IS NULL THEN
    RAISE EXCEPTION 'Not authenticated' USING ERRCODE = '42501';
  END IF;

  -- content_hash 형식 검증
  IF p_content_hash IS NULL OR char_length(p_content_hash) <> 64 THEN
    RAISE EXCEPTION 'Invalid content_hash (must be 64 hex chars)' USING ERRCODE = '22023';
  END IF;
  IF p_content_hash !~ '^[0-9a-f]{64}$' THEN
    RAISE EXCEPTION 'Invalid content_hash format' USING ERRCODE = '22023';
  END IF;

  IF NOT public.is_tenant_member(p_tenant_id) THEN
    RAISE EXCEPTION 'Not a tenant member' USING ERRCODE = '42501';
  END IF;

  -- approval_line 기본 검증
  IF jsonb_typeof(p_approval_line) <> 'array'
     OR jsonb_array_length(p_approval_line) = 0 THEN
    RAISE EXCEPTION 'Approval line must have at least one item' USING ERRCODE = '22023';
  END IF;

  -- 1차 순회: user_ids 수집 + approval_count + group_orders 수집
  v_index := 0;
  FOR v_item IN SELECT * FROM jsonb_array_elements(p_approval_line) LOOP
    v_uid := (v_item->>'userId')::uuid;
    IF v_uid = ANY(v_user_ids) THEN
      RAISE EXCEPTION 'Duplicate approver in line: %', v_uid USING ERRCODE = '22023';
    END IF;
    v_user_ids := v_user_ids || v_uid;

    IF (v_item->>'stepType') = 'approval' THEN
      v_approval_count := v_approval_count + 1;
    END IF;

    -- group_order: 지정 없으면 index 로 fallback (순차)
    v_group_order := COALESCE((v_item->>'groupOrder')::int, v_index);
    IF v_group_order < 0 THEN
      RAISE EXCEPTION 'groupOrder must be >= 0' USING ERRCODE = '22023';
    END IF;
    v_group_orders := v_group_orders || v_group_order;
    IF v_group_order > v_max_group THEN
      v_max_group := v_group_order;
    END IF;

    v_index := v_index + 1;
  END LOOP;

  IF v_approval_count = 0 THEN
    RAISE EXCEPTION 'At least one approval step required' USING ERRCODE = '22023';
  END IF;

  -- group_order 연속성 검증: distinct set 이 {0, 1, ..., N-1}
  IF (SELECT count(DISTINCT g) FROM unnest(v_group_orders) AS g) <> v_max_group + 1 THEN
    RAISE EXCEPTION 'groupOrder must be contiguous from 0 (no gaps)' USING ERRCODE = '22023';
  END IF;

  -- 각 group 에 최소 1개 approval step
  PERFORM 1
  FROM (
    SELECT gg.g
    FROM unnest(v_group_orders) WITH ORDINALITY AS u(g, ord)
    JOIN jsonb_array_elements(p_approval_line) WITH ORDINALITY AS s(item, ord) USING (ord)
    WHERE s.item->>'stepType' = 'approval'
    GROUP BY gg.g
  ) AS t;
  -- 위 쿼리는 복잡하므로 대안: 루프 2차 순회로 각 group 에 approval 존재 여부 확인

  -- (단순화) 2차 순회로 각 group 의 approval count 체크
  DECLARE
    v_group_approval_counts int[] := array_fill(0, ARRAY[v_max_group + 1]);
    v_i int := 1;
  BEGIN
    FOR v_item IN SELECT * FROM jsonb_array_elements(p_approval_line) LOOP
      IF (v_item->>'stepType') = 'approval' THEN
        v_group_order := COALESCE((v_item->>'groupOrder')::int, v_i - 1);
        v_group_approval_counts[v_group_order + 1] :=
          v_group_approval_counts[v_group_order + 1] + 1;
      END IF;
      v_i := v_i + 1;
    END LOOP;
    FOR v_i IN 1..v_max_group + 1 LOOP
      IF v_group_approval_counts[v_i] = 0 THEN
        RAISE EXCEPTION 'Group % has no approval step', v_i - 1 USING ERRCODE = '22023';
      END IF;
    END LOOP;
  END;

  -- tenant_members 멤버십 확인
  IF EXISTS (
    SELECT 1 FROM unnest(v_user_ids) AS uid
    WHERE uid NOT IN (
      SELECT user_id FROM public.tenant_members WHERE tenant_id = p_tenant_id
    )
  ) THEN
    RAISE EXCEPTION 'Some approvers are not tenant members' USING ERRCODE = '42501';
  END IF;

  -- attachments 의 sha256 누락 체크 (v1.2 제약)
  IF array_length(p_attachment_ids, 1) > 0 THEN
    SELECT count(*) INTO v_missing_attachments
    FROM public.approval_attachments
    WHERE id = ANY(p_attachment_ids)
      AND sha256 IS NULL;
    IF v_missing_attachments > 0 THEN
      RAISE EXCEPTION 'Some attachments are missing sha256' USING ERRCODE = '22023';
    END IF;
  END IF;

  -- 양식 FOR UPDATE + snapshot
  SELECT schema INTO v_schema
  FROM public.approval_forms
  WHERE id = p_form_id
    AND is_published = true
    AND (tenant_id = p_tenant_id OR tenant_id IS NULL)
  FOR UPDATE;

  IF v_schema IS NULL THEN
    RAISE EXCEPTION 'Form not found or unpublished' USING ERRCODE = '02000';
  END IF;

  v_doc_number := public.fn_next_doc_number(p_tenant_id);

  -- amount 추출 (v1.1 로직)
  BEGIN
    v_amount := (p_content->>'amount')::numeric;
  EXCEPTION WHEN OTHERS THEN
    v_amount := NULL;
  END;

  -- document INSERT / UPDATE (v1.2: content_hash 저장, current_step_index=0)
  IF p_document_id IS NULL THEN
    INSERT INTO public.approval_documents (
      tenant_id, doc_number, form_id, form_schema_snapshot,
      content, content_hash, drafter_id, status, current_step_index, submitted_at
    )
    VALUES (
      p_tenant_id, v_doc_number, p_form_id, v_schema,
      p_content, p_content_hash, v_user_id, 'in_progress', 0, now()
    )
    RETURNING * INTO v_doc;
  ELSE
    UPDATE public.approval_documents
    SET doc_number = v_doc_number,
        form_schema_snapshot = v_schema,
        content = p_content,
        content_hash = p_content_hash,
        status = 'in_progress',
        current_step_index = 0,
        submitted_at = now(),
        updated_at = now()
    WHERE id = p_document_id
      AND drafter_id = v_user_id
      AND status = 'draft'
      AND tenant_id = p_tenant_id
    RETURNING * INTO v_doc;

    IF v_doc.id IS NULL THEN
      RAISE EXCEPTION 'Draft not found or not owned' USING ERRCODE = '42501';
    END IF;
  END IF;

  -- step 스냅샷 INSERT + delegation 매칭 (v1.1 M11 유지)
  v_index := 0;
  FOR v_item IN SELECT * FROM jsonb_array_elements(p_approval_line) LOOP
    v_step_type := v_item->>'stepType';
    v_uid := (v_item->>'userId')::uuid;
    v_group_order := COALESCE((v_item->>'groupOrder')::int, v_index);
    v_step_status := 'pending';
    v_step_comment := NULL;
    v_delegation_rule := NULL;

    IF v_step_type = 'approval' THEN
      SELECT dr.* INTO v_delegation_rule
        FROM public.delegation_rules dr
       WHERE dr.tenant_id = p_tenant_id
         AND dr.delegator_user_id = v_uid
         AND (dr.form_id = p_form_id OR dr.form_id IS NULL)
         AND (dr.amount_limit IS NULL
              OR (v_amount IS NOT NULL AND v_amount <= dr.amount_limit))
         AND dr.effective_from <= now()
         AND (dr.effective_to IS NULL OR dr.effective_to > now())
       ORDER BY
         (dr.form_id IS NULL) ASC,
         dr.amount_limit ASC NULLS LAST,
         dr.effective_from DESC
       LIMIT 1;

      IF v_delegation_rule.id IS NOT NULL THEN
        v_step_status := 'skipped';
        v_step_comment := format(
          '[전결] delegated to %s (rule %s)',
          v_delegation_rule.delegate_user_id,
          v_delegation_rule.id
        );
      END IF;
    END IF;

    INSERT INTO public.approval_steps (
      tenant_id, document_id, step_index, group_order, step_type,
      approver_user_id, status, acted_at, comment
    )
    VALUES (
      p_tenant_id, v_doc.id, v_index, v_group_order,
      v_step_type::public.step_type,
      v_uid,
      v_step_status,
      CASE WHEN v_step_status = 'skipped' THEN now() ELSE NULL END,
      v_step_comment
    );

    IF v_delegation_rule.id IS NOT NULL THEN
      INSERT INTO public.approval_audit_logs (
        tenant_id, document_id, actor_id, action, payload
      ) VALUES (
        p_tenant_id, v_doc.id, v_user_id, 'delegated',
        jsonb_build_object(
          'step_index', v_index,
          'group_order', v_group_order,
          'original_approver', v_uid,
          'delegate_to', v_delegation_rule.delegate_user_id,
          'rule_id', v_delegation_rule.id,
          'amount', v_amount
        )
      );
    END IF;

    v_index := v_index + 1;
  END LOOP;

  -- 첨부 확정
  IF array_length(p_attachment_ids, 1) > 0 THEN
    UPDATE public.approval_attachments
    SET document_id = v_doc.id
    WHERE id = ANY(p_attachment_ids)
      AND uploaded_by = v_user_id
      AND document_id IS NULL
      AND tenant_id = p_tenant_id;
  END IF;

  -- 상신 audit (content_hash mirror to payload)
  INSERT INTO public.approval_audit_logs (
    tenant_id, document_id, actor_id, action, payload
  ) VALUES (
    p_tenant_id, v_doc.id, v_user_id, 'submit',
    jsonb_build_object(
      'content_hash', p_content_hash,
      'approval_line', p_approval_line,
      'attachment_ids', p_attachment_ids
    )
  );

  -- 첫 그룹으로 advance (delegation 으로 일부 skipped 된 경우 자동 처리)
  v_doc := public.fn_advance_document(v_doc.id);

  RETURN v_doc;
END;
$$;
```

## 6. JCS + Hash Module Design

### 6.1 `src/lib/hash/jcs.ts` (RFC 8785)

RFC 8785 핵심 규칙:
1. Object key 는 UTF-16 code unit 기준 정렬
2. Number 는 ECMA-262 `Number.prototype.toString()` 변환 후 JSON Serializer 규칙 재적용 (15자리 dtoa, exponent 제거 등)
3. String 은 JSON 의 최소 escape rule (double quote, backslash, control chars < U+0020)
4. null/bool 은 기본 JSON 형태
5. Array 순서 유지

v1.2 범위에서는 Rigel 의 content 가 **string/number/boolean/null/array/object 조합만** 사용 (Rigel form schema 상 제약), BigInt/Infinity/NaN 은 나타나지 않음. 자체 구현 ~80줄 목표:

```typescript
// src/lib/hash/jcs.ts
// RFC 8785 JSON Canonicalization Scheme — Rigel 제한 사양
// 미지원: BigInt, Infinity, NaN (Rigel form schema 상 생성 불가)
// 지원: string, number (ECMA-262 Number.toString), boolean, null, array, object

export function canonicalize(value: unknown): string {
  return serialize(value);
}

function serialize(v: unknown): string {
  if (v === null) return 'null';
  if (v === undefined) throw new Error('JCS: undefined not allowed');
  if (typeof v === 'boolean') return v ? 'true' : 'false';
  if (typeof v === 'number') return serializeNumber(v);
  if (typeof v === 'string') return serializeString(v);
  if (Array.isArray(v)) {
    return '[' + v.map(serialize).join(',') + ']';
  }
  if (typeof v === 'object') {
    const obj = v as Record<string, unknown>;
    // UTF-16 code unit 정렬
    const keys = Object.keys(obj).sort((a, b) => {
      // JS string compare 는 이미 UTF-16 code unit 순서
      return a < b ? -1 : a > b ? 1 : 0;
    });
    const parts = keys.map(
      (k) => serializeString(k) + ':' + serialize(obj[k])
    );
    return '{' + parts.join(',') + '}';
  }
  throw new Error(`JCS: unsupported type ${typeof v}`);
}

function serializeNumber(n: number): string {
  if (!Number.isFinite(n)) {
    throw new Error(`JCS: non-finite number ${n}`);
  }
  if (Object.is(n, -0)) return '0';  // -0 → 0
  // Number.prototype.toString 이 ECMA-262 준수 ("1", "1.5", "1.5e+21", ...)
  // RFC 8785 는 이를 base 로 JSON Serializer 규칙 적용 — 여기서는 실질적으로 toString 결과 그대로 OK
  // (단, 1e21 이상 exponent 표기는 JSON 이 허용하므로 그대로 유지)
  return n.toString();
}

function serializeString(s: string): string {
  let out = '"';
  for (let i = 0; i < s.length; i++) {
    const c = s.charCodeAt(i);
    if (c === 0x22) out += '\\"';
    else if (c === 0x5c) out += '\\\\';
    else if (c === 0x08) out += '\\b';
    else if (c === 0x09) out += '\\t';
    else if (c === 0x0a) out += '\\n';
    else if (c === 0x0c) out += '\\f';
    else if (c === 0x0d) out += '\\r';
    else if (c < 0x20) out += '\\u' + c.toString(16).padStart(4, '0');
    else out += s[i];
  }
  return out + '"';
}
```

### 6.2 `src/lib/hash/documentHash.ts`

```typescript
// src/lib/hash/documentHash.ts
// v1.2: content_hash 계산. 클라이언트·서버 양쪽 호환.

import { canonicalize } from './jcs';
import type { FormSchema, ApprovalLineItem } from '$lib/types/approval';

export interface DocumentHashInput {
  formSchema: FormSchema;
  content: Record<string, unknown>;
  approvalLine: ApprovalLineItem[];
  attachmentSha256s: string[];  // sort 된 상태로 전달
}

export async function computeDocumentHash(input: DocumentHashInput): Promise<string> {
  // 해시 입력 = 고정 스키마 형태의 객체를 JCS 로 canonicalize 후 SHA-256
  const normalized = {
    schema: input.formSchema,
    content: input.content,
    line: input.approvalLine.map((i) => ({
      userId: i.userId,
      stepType: i.stepType,
      groupOrder: i.groupOrder ?? null
    })),
    attachments: [...input.attachmentSha256s].sort()
  };
  const canonical = canonicalize(normalized);
  return sha256Hex(canonical);
}

async function sha256Hex(input: string): Promise<string> {
  // Browser: Web Crypto API
  // Node: 같은 API (undici/global crypto in Node 20+)
  const buffer = new TextEncoder().encode(input);
  const digest = await crypto.subtle.digest('SHA-256', buffer);
  return Array.from(new Uint8Array(digest))
    .map((b) => b.toString(16).padStart(2, '0'))
    .join('');
}
```

## 10. Security Considerations

- [ ] **RLS**: 0031~0036 apply 후 `get_advisors` — Security/Performance 0 WARN
- [ ] **Storage RLS**: `user-signatures` 4개 정책 (insert/update/select/delete) 확인, `FOR ALL` 없음
- [ ] **본인만 서명 upsert**: `storage.foldername(name)[1] = auth.uid()`
- [ ] **tenant 멤버만 서명 read**: `EXISTS` subquery 로 `tenant_members` 동일 tenant 검증
- [ ] **content_hash 필수 검증** RPC 초입 (Node 측 우회 불가)
- [ ] **서명 PNG magic bytes** 서버 측 재검증 (클라 검증 우회 차단)
- [ ] **서명 sha256 snapshot** audit_logs.payload 에 기록 — 사후 서명 교체에도 증거 보존
- [ ] **PDF self-request cookie** — 같은 origin, HTTPS 시 secure cookie 유지. non-HTTPS dev 에서도 작동.
- [ ] **puppeteer `--no-sandbox`** — root 로 실행할 때만 필요. non-root 시 제거 권장 (Dockerfile 에서 non-root user 로 실행 시)
- [ ] **A11y**: 병렬 그룹 체크박스 label 연결, svelte-check `a11y_*` 0 warning

## 14. Clean Architecture Mapping

Rigel 은 Dynamic level (SvelteKit + Supabase 공용 SaaS 패턴) — 순수 Clean Architecture 가 아닌 **SvelteKit 관용 모듈 구조**. v1.2 에서도 동일:

| Layer | Location | v1.2 추가 |
|---|---|---|
| Presentation | `src/routes/**/+page.svelte`, `src/lib/components/` | DocumentSteps 업그레이드, ApprovalLine 그룹 UI, signature 페이지 |
| Application | `src/routes/**/+page.server.ts` (actions) | drafts/new submit action 에 해시 계산, pdf 라우트, 서명 업로드 action |
| Domain | `src/lib/types/approval.ts`, `src/lib/server/schemas/*` | 타입 확장, Zod schema 에 groupOrder 필드 |
| Infrastructure | `src/lib/server/*`, `src/lib/client/*`, Supabase client | hash 모듈, pdf renderer, uploadAttachment sha256 |
| DB Layer | `supabase/migrations/*` | 0031~0036 |

**의존성 방향**: Route → Components → types / server queries → Supabase. v1.0/v1.1 와 동일.

## 15. Coding Convention (v1.0/v1.1 유지 + v1.2 추가)

### 15.1 기존 유지 (CLAUDE.md 참조)

- RLS: `(SELECT auth.uid())`, FOR ALL 금지, action 별 분리
- Mutation: SECURITY DEFINER RPC only
- Enum 추가 시 TX 외부 분리 (v1.2 는 enum 추가 없음)
- Svelte 5: `untrack` for `$state(data.x)`, hidden input primitive only
- `$lib/server/*` → `.svelte` import 금지

### 15.2 v1.2 신규 관행

| 항목 | 규칙 |
|---|---|
| **content_hash** | RPC 에서 형식 (64 hex) 강제. Node 측 계산만 허용. |
| **sha256 (attachments)** | 클라이언트 `crypto.subtle.digest` 로 계산. submit 전 NOT NULL 보장. |
| **Signature path** | Storage `user-signatures` 버킷, path `{user_id}/signature.png` 고정 |
| **audit payload snapshot** | 승인 시 `signature_storage_path`, `signature_sha256` 필수 포함 |
| **PDF 모드** | `?print=1` 쿼리만 허용. CSS `@media print` 사용하지 않음 (명시 prop 제어가 우월) |
| **fn_advance_document 의무 호출** | 모든 step 상태 변경 RPC 의 **마지막** 작업으로 PERFORM |
| **group_order** | RPC 파라미터 명: `groupOrder` (camelCase). DB 컬럼: `group_order` (snake_case). |

## 18. Rollback Strategy

Forward-only 가 원칙이지만, 심각한 장애 시 다음 SQL 로 v1.1 상태 복구 가능:

```sql
-- 0036 → 0024 + 0026 순서로 REPLACE
CREATE OR REPLACE FUNCTION public.fn_approve_step(...) AS $$ -- 0024 의 body $$;
CREATE OR REPLACE FUNCTION public.fn_reject_step(...) AS $$ -- 0024 의 body $$;
CREATE OR REPLACE FUNCTION public.fn_submit_draft(...) AS $$ -- 0021 의 body $$;
-- 0035 fn_submit_post_facto 는 0026 으로 복구

-- 컬럼 drop (optional, 데이터 손실)
ALTER TABLE public.approval_documents DROP COLUMN content_hash;
ALTER TABLE public.approval_attachments DROP COLUMN sha256;
ALTER TABLE public.profiles DROP COLUMN signature_storage_path, DROP COLUMN signature_sha256;
-- group_order 는 drop 해도 step_index 와 값 동일 → 회귀 안전
ALTER TABLE public.approval_steps DROP COLUMN group_order;

-- Storage 버킷·정책은 Supabase Studio 에서 수동 삭제
```

**중요**: Rollback 전 `pg_dump` 필수. 프로덕션은 migration forward-only 를 고수.
