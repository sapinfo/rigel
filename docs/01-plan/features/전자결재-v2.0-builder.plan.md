# Plan Plus — 전자결재-v2.0-builder

## Executive Summary

| 관점 | 내용 |
|---|---|
| **Problem** | v1.x까지 양식은 JSON schema 수동 편집, 결재선은 기안자가 매번 수동 구성 → 관리자의 IT 의존도 높음, 기안자 반복 작업 |
| **Solution** | 드래그&드롭 양식 빌더(svelte-dnd-action) + 조직도 기반 결재선 자동 구성 + 양식 버전 관리 + 조건부 필드 로직 |
| **Function / UX Effect** | 관리자가 코드 없이 GUI로 양식 설계·발행 → 기안자가 부서/금액 조건에 따라 결재선 자동 부여 → 양식 수정 시 기존 문서 영향 없는 버전 관리 |
| **Core Value** | **"관리자 자립"** — IT 의존 없이 양식/결재선 운영. SMB SaaS의 셀프서비스 완성 → 고객 온보딩 비용 제거 |

## 1. User Intent Discovery

### 1.1 Core Problem
현재 결재 양식 생성은 `form_schema` JSON을 직접 작성해야 하며, 결재선은 기안자가 매번 수동으로 구성. 관리자가 비개발자인 SMB 환경에서 이는 도입 장벽.

### 1.2 Target Users
- **관리자** (primary): 양식 설계 + 결재선 규칙 설정
- **기안자**: 양식 작성 시 자동 결재선 + 조건부 필드 체험
- **IT 담당자**: 기존 JSON schema에서 빌더로 전환

### 1.3 Success Criteria
- **비개발자 양식 생성**: 관리자가 코드 없이 GUI로 양식 생성·발행
- **결재선 자동 구성**: 부서/직급 규칙으로 기안 시 자동 결재선
- **버전 무결성**: 양식 수정 시 기존 상신된 문서에 영향 없음
- **조건부 필드**: 특정 값에 따라 필드 표시/숨김

## 4. Feature Scope

### 4.1 양식 빌더

**라우트**: `/t/{slug}/admin/forms/builder/[formId]`

**UI 구조**:
- **좌측 사이드바**: 필드 팔레트 (text, textarea, number, date, date-range, select, attachment)
- **중앙 캔버스**: D&D 영역. 필드 추가/삭제/순서변경. 각 필드 클릭 시 속성 편집 (label, required, options 등)
- **우측 패널**: 실시간 미리보기 (기존 FormRenderer 재사용)
- **상단 툴바**: 저장(draft) / 미리보기 / 발행 / 버전 히스토리

**데이터 흐름**:
```
빌더 UI → $state(fields: FormField[]) → JSON schema 변환 → FormRenderer 미리보기
         → "발행" 클릭 → approval_forms INSERT (또는 새 version)
```

**기존 호환**: 빌더가 생성하는 schema 포맷은 기존 `FormSchema` 타입과 100% 호환. FormRenderer 변경 없음.

### 4.2 결재선 빌더

**신규 테이블**:
```
departments (id, tenant_id, name, parent_id, sort_order)
job_titles (id, tenant_id, name, level)  -- level: 1(사원) ~ 10(대표)
approval_rules (id, tenant_id, form_id?, condition_json, line_template_json, priority)
```

**profiles 확장**:
```
profiles ADD COLUMN department_id uuid REFERENCES departments
profiles ADD COLUMN job_title_id uuid REFERENCES job_titles
```

**condition_json 구조**:
```json
{ "field": "amount", "operator": "gte", "value": 1000000 }
```

**line_template_json 구조**:
```json
[
  { "role": "team_lead", "stepType": "approval" },
  { "role": "department_head", "stepType": "approval" },
  { "role": "ceo", "stepType": "approval", "condition": { "field": "amount", "operator": "gte", "value": 5000000 } }
]
```

**동작**: 기안 시 `fn_submit_draft`에서 결재선이 비어있으면 → `approval_rules` 매칭 → 기안자의 부서/직급에서 상위 결재자 자동 탐색 → `approval_steps` 자동 생성.

### 4.3 양식 버전 관리

**approval_forms 확장**:
```
approval_forms ADD COLUMN version int DEFAULT 1
approval_forms ADD COLUMN parent_form_id uuid REFERENCES approval_forms(id)
```

**동작**:
- 발행된 양식 수정 시 → 새 row (parent_form_id = 원본, version = 원본+1)
- 기존 문서는 원 form_id 유지 → `form_schema_snapshot`으로 이미 보호됨
- 양식 목록은 `parent_form_id IS NULL OR id = latest` 필터

### 4.4 조건부 필드

**form_schema 확장**:
```json
{
  "fields": [...],
  "conditionals": [
    {
      "if": { "fieldId": "type", "operator": "eq", "value": "출장" },
      "show": ["destination", "travel_dates"],
      "hide": ["office_location"]
    }
  ]
}
```

**FormRenderer 변경**: `conditionals` 배열 존재 시 → 각 필드의 visible 상태를 `$derived`로 계산 → hidden 필드는 DOM에서 제거 + submit 시 값 제외.
