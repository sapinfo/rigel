# Plan Plus — 전자결재-v1.3-realtime-notification

## Executive Summary

| 관점 | 내용 |
|---|---|
| **Problem** | v1.2까지 결재 시스템의 기능은 완결되었으나, 결재 요청/승인/반려 발생 시 사용자가 직접 새로고침해야 확인 가능 → 결재 응답 시간 지연, 긴급 결재 누락, 모바일 접근 불가 |
| **Solution** | Supabase Realtime (Postgres Changes) + Edge Function 이메일 + 반응형 CSS + PWA 홈스크린 설치. DB trigger 기반으로 audit_logs INSERT 시 자동 알림 생성 + 이메일 발송 |
| **Function / UX Effect** | 결재자가 브라우저 알림 bell로 즉시 인지 → 이메일로 이중 도달 보장 → 모바일 홈스크린에서 1-tap 접근 → 결재 응답 시간 50%+ 단축 |
| **Core Value** | **"놓치지 않는 결재"** 달성. 실시간 + 이메일 이중 채널로 알림 누락 제로, 모바일 접근으로 장소 제약 해소 → SMB 도입 장벽 최종 제거 |

## 1. User Intent Discovery

### 1.1 Core Problem
결재 시스템이 "push" 없이 "pull" 방식으로만 동작 → 결재자가 직접 inbox를 확인하지 않으면 결재 요청을 모름. 결재 리드타임의 병목은 기능이 아닌 **인지 지연**.

### 1.2 Target Users
- **결재자** (primary): 결재 요청 즉시 인지 → 신속 승인/반려
- **기안자**: 결재 완료/반려 결과 즉시 확인
- **관리자**: 장기 미결 문서 현황 파악
- **이동 중 사용자**: 모바일 브라우저/PWA로 결재 처리

### 1.3 Success Criteria
- **결재 응답 시간 50%+ 단축**: 실시간 알림으로 결재자 인지 지연 제거
- **알림 누락 제로**: Realtime + 이메일 이중 채널
- **모바일 결재 가능**: 반응형 UI + PWA 홈스크린 설치

### 1.4 Constraints
- Supabase local 환경에서 Edge Function 테스트 가능해야 함 (`supabase functions serve`)
- 이메일 SMTP는 환경변수 기반 (dev: console 로그, prod: 실제 발송)
- PWA는 HTTPS 필수 (dev: localhost 예외)
- 기존 v1.0~v1.2 코드 변경 최소화 (additive only)

## 2. Alternatives Explored

### D1: 이메일 발송 방식

| 옵션 | 설명 | 선택 |
|---|---|---|
| **D1-A: Edge Function + SMTP** | DB trigger → pg_net → Edge Function → Resend/SendGrid. Supabase 스택 내 완결 | **선택** |
| D1-B: pg_net 직접 HTTP | DB 내부에서 pg_net으로 SendGrid API 직접 호출. 디버깅 어려움 | - |
| D1-C: SvelteKit nodemailer | adapter-node 의존, 폴링 방식 | - |

**이유**: Edge Function은 이메일 템플릿을 TypeScript로 관리 가능, 타입 안전, 로컬 테스트(`supabase functions serve`) 지원.

### D2: 실시간 구독 대상

| 옵션 | 설명 | 선택 |
|---|---|---|
| **D2-A: notifications 테이블 구독** | 전용 테이블 → RLS로 본인 알림만 구독 | **선택** |
| D2-B: audit_logs 직접 구독 | 기존 테이블 활용이나 RLS 복잡, 불필요한 이벤트 과다 | - |

**이유**: notifications 전용 테이블이 RLS 단순화 + 읽음 상태 관리 용이.

### D3: 모바일 접근 방식

| 옵션 | 설명 | 선택 |
|---|---|---|
| **D3-A: 반응형 CSS + PWA manifest** | Tailwind breakpoints + manifest.json 홈스크린 설치 | **선택** |
| D3-B: React Native / Flutter | 별도 앱 개발. scope 과다 | - |
| D3-C: Capacitor wrapper | SvelteKit을 네이티브 앱으로 래핑. 복잡도 증가 | - |

**이유**: SMB SaaS에 네이티브 앱은 과도. PWA로 충분한 모바일 경험 제공.

## 4. Feature Scope

### 4.1 Notifications 시스템

**notifications 테이블**:
```
notifications (
  id uuid PK DEFAULT gen_random_uuid(),
  tenant_id uuid FK NOT NULL,
  user_id uuid FK NOT NULL,      -- 수신자
  document_id uuid FK NOT NULL,
  type notification_type NOT NULL, -- 'approval_requested' | 'approved' | 'rejected' | 'commented' | 'withdrawn'
  title text NOT NULL,
  body text,
  read boolean DEFAULT false,
  created_at timestamptz DEFAULT now()
)
```

**알림 유형별 수신자**:

| 이벤트 | audit_action | 알림 수신자 | 알림 type |
|---|---|---|---|
| 상신 (submit) | `submit` | 첫 번째 결재자 (또는 현재 그룹) | `approval_requested` |
| 승인 (approve) | `approve` / `approved_by_proxy` | 다음 결재자 + 기안자 | `approval_requested` (다음), `approved` (기안자) |
| 최종 승인 | `approve` (마지막) | 기안자 | `approved` |
| 반려 | `reject` | 기안자 | `rejected` |
| 코멘트 | `comment` | 기안자 + 모든 결재자 | `commented` |
| 회수 | `withdraw` | 현재 결재자 | `withdrawn` |

**DB trigger**: `fn_create_notification()` — `audit_logs` AFTER INSERT → notifications INSERT + Edge Function 호출

### 4.2 Realtime 구독

- `+layout.svelte` (tenant scope)에서 `supabase.channel('notifications')` 구독
- `postgres_changes` filter: `event=INSERT, schema=public, table=notifications, filter=user_id=eq.{userId}`
- 새 알림 수신 시 `$state` unread count 업데이트 + 드롭다운 리스트 prepend

### 4.3 이메일 알림

- **Edge Function**: `supabase/functions/send-notification-email/index.ts`
- **호출**: DB trigger에서 `pg_net.http_post()` → Edge Function URL
- **템플릿**: 알림 유형별 한국어 이메일 (제목/본문)
- **환경변수**: `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM`
- **Dev mode**: SMTP 미설정 시 console.log로 fallback

### 4.4 인앱 알림 UI

- **NotificationBell.svelte**: header 우측에 bell icon + unread count 배지
- **NotificationDropdown.svelte**: 최근 20건 표시, 클릭 시 해당 문서로 이동 + read=true
- **Mark all read**: 전체 읽음 처리 버튼
- **Empty state**: "새로운 알림이 없습니다"

### 4.5 반응형 레이아웃

**Breakpoints** (Tailwind):
- `sm` (640px): 모바일
- `md` (768px): 태블릿
- `lg` (1024px): 데스크톱 (현재)

**변경 대상 컴포넌트**:

| 컴포넌트 | 모바일 변경 |
|---|---|
| 레이아웃 header | 햄버거 메뉴 (nav collapse) |
| inbox 탭 | 가로 스크롤 또는 드롭다운 |
| document detail | 단일 컬럼 스택 |
| ApprovalPanel | 하단 고정 (sticky bottom) |
| ApprovalLine (편집) | 세로 리스트 (체크박스 유지) |
| FormRenderer | 단일 컬럼 (기존 그리드 해제) |

### 4.6 PWA

- `static/manifest.json`: name, icons, start_url, display: standalone, theme_color
- `src/service-worker.ts`: SvelteKit 기본 service worker (캐시 전략: network-first)
- `app.html`: `<link rel="manifest">` + `<meta name="theme-color">`
- Icons: 192x192, 512x512 PNG

## 6. Risk Table

| # | 위험 | 영향 | 완화 |
|---|---|---|---|
| R20 | Supabase Realtime 연결 수 제한 (200 concurrent) | SMB 환경에서 충분하나 대규모 tenant 시 초과 가능 | 연결 수 모니터링 + 알림. v2에서 connection pooling 검토 |
| R21 | Edge Function cold start (~200ms) | 이메일 발송 지연 | fire-and-forget. 사용자는 인앱 알림으로 즉시 인지 |
| R22 | pg_net extension 미설치 | Edge Function 호출 불가 | migration에서 `CREATE EXTENSION IF NOT EXISTS pg_net` |
| R23 | SMTP 미설정 시 이메일 실패 | dev/staging에서 이메일 없음 | console.log fallback + 환경변수 존재 체크 |
| R24 | PWA HTTPS 필수 | localhost dev에서는 예외, staging에서 필요 | SvelteKit adapter-node + reverse proxy (nginx/caddy) |
| R25 | 반응형 레이아웃 회귀 | 기존 데스크톱 UI 깨짐 | 데스크톱 breakpoint 기본값 유지, mobile-first 추가만 |
| R26 | notifications 테이블 대량 데이터 | 오래된 알림 누적 | 90일 이상 read=true 알림 자동 삭제 (pg_cron) |
