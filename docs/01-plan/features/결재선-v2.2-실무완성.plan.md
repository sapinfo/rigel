## Executive Summary

| 항목 | 내용 |
|---|---|
| **Feature** | 한국형 결재 실무 완성 — 즐겨찾기, 일괄결재, 합의, 재기안, 독촉, 대시보드 |
| **선행 버전** | v2.1 (match 93%) — 결재선 관리 GUI 완성 |
| **Migration** | 0049~ |
| **핵심 목표** | ApprovalAddin에 있으나 Rigel에 없는 실무 필수 기능 11개 추가 |
| **성공 기준** | 기안→결재 전 과정에서 경쟁 제품 대비 기능 격차 해소 |

### Value Delivered (4관점)

| 관점 | 내용 |
|---|---|
| **Problem** | Rigel은 결재 핵심 흐름(기안→승인/반려)은 갖췄지만, 실무에서 매일 쓰는 기능(즐겨찾기, 일괄결재, 재기안, 독촉, 대시보드)이 없어 실사용 불가 |
| **Solution** | (1) 결재선 즐겨찾기+양식별 템플릿, (2) 일괄결재, (3) 합의(agree) step, (4) 재기안/독촉, (5) 긴급 문서, (6) StampLine, (7) 대시보드, (8) 감사로그 뷰어 |
| **Function UX** | 기안자: 즐겨찾기 1클릭 결재선 → 긴급 표시 → 상신. 결재자: 대시보드에서 미결 확인 → 체크박스 일괄 승인. 관리자: 감사 로그 조회 |
| **Core Value** | "두레이·더존에서 넘어온 사용자가 낯설지 않은 UX" = 한국형 그룹웨어 경쟁력 확보 |

- 병렬 결재 OR 조건 — 현재 AND만
- 보관기한/보안등급 메타데이터

## 2. Feature Details

### M1: 결재선 즐겨찾기

**신규 테이블**: `approval_line_favorites`

```
id (uuid PK), tenant_id (FK), user_id (FK), name (text),
line_json (jsonb — ApprovalLineItem[]), form_id (nullable FK),
created_at (timestamptz)
```

**동작**:
- 기안 페이지에서 결재선 구성 후 "즐겨찾기 저장" 버튼
- 즐겨찾기 목록에서 1클릭으로 결재선 불러오기
- 양식별 필터 가능 (form_id가 같은 즐겨찾기 우선 표시)
- 삭제 가능

**UI 위치**: `drafts/new/[formCode]/+page.svelte` — 결재선 섹션 상단

```
┌─────────────────────────────────────────────────┐
│ ★ 즐겨찾기                            [저장]     │
│   ├─ 일반 품의 (김과장→이부장→홍대표)    [적용]   │
│   └─ 경비 청구 (최재무→홍대표)           [적용]   │
│                                                  │
│ 📋 추천 결재선 (관리자 설정)                      │
│   └─ 기본: 팀장→부서장→대표              [적용]   │
│                                                  │
│ 🤖 자동 구성 (조직도 기반)               [구성]   │
│                                                  │
│ 결재선 ─────────────────────────────────────────  │
│  1. 김과장 (영업1팀/과장)  승인   [×]            │
│  2. 홍대표 (경영지원/대표)  승인   [×]            │
└─────────────────────────────────────────────────┘
```

### M2: 결재선 템플릿 (양식별 추천)

**신규 테이블**: `approval_line_templates`

```
id (uuid PK), tenant_id (FK), name (text), form_id (nullable FK),
line_json (jsonb), is_default (boolean), priority (int),
created_by (FK), created_at (timestamptz)
```

**동작**:
- 관리자가 양식별 또는 전체 공통 결재선 템플릿을 등록
- `is_default=true`인 템플릿은 기안 시 자동 적용
- 기존 `approval_rules` (자동 해석)와 공존 — 우선순위: 즐겨찾기 > 템플릿 > 자동해석

**관리 UI**: `/admin/approval-templates` 새 라우트

### M3: 일괄 결재

**동작**:
- 인박스 pending 탭에서 체크박스로 여러 문서 선택
- "일괄 승인" / "일괄 반려" 버튼
- 확인 다이얼로그 (코멘트 입력 선택)
- 서버: 선택된 document_id 배열 → 루프로 fn_approve_step / fn_reject 호출
- 실패 건은 에러 목록으로 반환

**신규 action**: `batchApprove`, `batchReject` in inbox `+page.server.ts`

**UI**: `inbox/+page.svelte` — pending 탭에 체크박스 + 일괄 액션 바

### M4: 합의 (agree) step type

**현재**: `StepType = 'approval' | 'reference'`
**변경**: `StepType = 'approval' | 'agreement' | 'reference'`

- `agreement` step: 결재선에서 합의 역할. disagreed 시 문서 전체 rejected
- DB: `step_type` 컬럼은 text → enum 변경 불필요 (기존 text 호환)
- fn_approve_step에 `agreement` 처리 분기 추가
- UI: ApprovalPanel에 "동의/부동의" 버튼 추가

### M5: 재기안

**동작**:
- 반려된 문서 상세에서 "재기안" 버튼
- 기존 문서의 content, form, 결재선을 복사하여 새 draft 생성
- 수정 후 다시 상신 가능

**구현**: `documents/[id]/+page.server.ts`에 `resubmit` action 추가
- fn_save_draft RPC로 새 문서 생성 (기존 content 복사)
- redirect → 새 문서 편집 페이지

### M6: 결재 독촉

**동작**:
- 상신한 문서에서 현재 결재자에게 "독촉" 버튼
- Supabase Realtime 또는 notifications 테이블에 알림 삽입
- 기존 v1.3 알림 인프라 재사용

**구현**: `documents/[id]/+page.server.ts`에 `remind` action 추가

### M7: 긴급 문서

**DB 변경**: `approval_documents`에 `urgency` 컬럼 추가 (`text DEFAULT '일반'`)
- 값: `'일반'` | `'긴급'`

**UI**:
- 기안 페이지: 긴급 토글/셀렉트
- 인박스: 긴급 문서에 빨간 배지
- 대시보드: 긴급 미결 별도 카운트

### M8: 대시보드

**신규 라우트**: `/t/[tenantSlug]/dashboard` (또는 기존 `/t/[tenantSlug]` 메인)

```
┌──────────────────────────────────────────────┐
│ 좋은 아침입니다, 김인사님                      │
│ ────────────────────────────────────────────  │
│ [미결 5] [상신진행 3] [승인완료 12]            │
│ [반려 1] [임시저장 2] [전체 23]                │
│                                               │
│ ★ 즐겨찾기 결재선                              │
│   ├─ 일반 품의 → [새 기안]                     │
│   └─ 경비 청구 → [새 기안]                     │
│                                               │
│ 부재 현황                                      │
│   ├─ 이인사 (인사팀) → 대결: 김인사             │
│   └─ 서영업 (영업2팀) → 대결: 윤영업            │
└──────────────────────────────────────────────┘
```

**데이터**: 기존 inbox load 쿼리 재사용 (count만 추출) + absences 조회

### M9: StampLine (가로 도장)

**컴포넌트**: `StampLine.svelte`

```
┌──────┬──────┬──────┬──────┐
│ 기안  │ 검토  │ 승인  │ 최종  │
├──────┼──────┼──────┼──────┤
│ 박인사 │ 김인사 │ 홍대표 │      │
│ 사원  │ 부장  │ 대표  │      │
│  ✅  │  ✅  │ 대기  │      │
└──────┴──────┴──────┴──────┘
```

- 문서 상세 페이지 상단에 가로 도장 표시
- 각 step의 상태: 대기/승인/반려/합의/참조 아이콘
- 기존 ApprovalStep 데이터 기반 렌더링

### M10: 감사 로그 뷰어

**신규 라우트**: `/t/[tenantSlug]/admin/audit`

- audit_logs 테이블 조회 (기존 v1.0부터 INSERT됨)
- 페이지네이션 (50건씩)
- 필터: 날짜 범위, action 유형, 사용자
- 관리자 전용 (admin/owner)

### M11: OrgTreePicker

**컴포넌트**: `OrgTreePicker.svelte`

- 기존 ApproverPicker (flat 멤버 목록) 대체/보강
- 부서 트리에서 멤버 선택
- 접기/펼치기
- 검색 필터
- 결재선 편집 시 사용 (ApprovalLine 컴포넌트와 연동)

## 5. Risk & Mitigation

| # | 리스크 | 영향 | 대응 |
|---|---|---|---|
| R1 | 일괄 결재 시 partial failure | High | 트랜잭션 단위 처리 + 실패 건 목록 반환 |
| R2 | agree step type 추가 시 기존 문서 호환 | Medium | 기존 문서는 approval/reference만 사용 → 하위호환 유지 |
| R3 | 즐겨찾기 line_json에 탈퇴 멤버 포함 | Medium | 적용 시 userId 유효성 검사 → 없는 멤버 경고 |
| R4 | 대시보드 쿼리 성능 (count 6개) | Low | 단일 쿼리로 status별 GROUP BY |
| R5 | StampLine 반응형 (모바일) | Low | 가로 스크롤 또는 세로 전환 |
