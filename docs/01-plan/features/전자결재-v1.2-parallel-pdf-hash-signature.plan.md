## 1. User Intent Discovery (Plan Plus Phase 1)

### 1.1 Core Problem

v1.1 까지 **순차/예외 도메인**은 완성됐지만, 실제 SMB 운영에서 두 종류의 블로커가 잔존:

1. **다부서 동시 검토의 자연스러운 표현 불가** — 현재 순차 승인만 지원해 "재무팀 + 법무팀이 동시에 리뷰한 뒤 임원 결재" 같은 구조를 만들면 한쪽이 다른 쪽을 기다리며 리드타임 2배. 실무는 pseudo-parallel 을 순차로 꾸역꾸역 구성 중.
2. **사후 증빙 3종 (PDF / 해시 / 서명) 부재** — 승인된 결재 문서의 **외부 출력물** (감사, 법무 보관, 공공 납품) 이 불가능. "우리 시스템은 완결됐는데 *종이로 뽑을 수가 없다*" 라는 실사용자 피드백.

v1.2 는 위 두 블로커를 **하나의 feature** 로 묶는다. 이유: PDF 출력은 병렬 결재선을 표현할 UI 가 있어야 렌더 의미가 있고, 해시와 서명이미지는 PDF 에 함께 embed 되어야 "증빙 문서" 로서의 일관성을 가짐. 잘라서 내면 세 번의 반쪽짜리 릴리스가 됨.

### 1.2 Target Users

| User Type | Usage Context | Key Need |
|---|---|---|
| **기안자** | 다부서 병렬 검토가 필요한 기안 (예: 계약서 — 법무+재무+구매 동시 검토) | 드래그로 병렬 그룹 구성 |
| **결재자** | 본인 차례가 되었을 때 인박스 표시 | 본인 step 이 어떤 그룹에 속하는지 시각 표시, 승인 시 자동 서명 embed |
| **감사자 / 법무** | 사후에 결재 문서의 정당성 증명 필요 | PDF 1-click + content_hash 로 변조 검증 |
| **임원** | 종이 결재 관행의 디지털 잔상 — "내 사인이 찍혀 나오는" 출력물 | 서명이미지 자동 embed, 한 번 등록 후 재사용 |
| **멀티 테넌트 오너 (Rigel SaaS 고객사)** | 감사·법무 요건을 갖춘 SMB | "인쇄 가능한 결재" 라는 B2B sell 지표 달성 |

- [ ] **병렬결재 smoke 4건** 통과: S40 순차 + 병렬 혼합 / S41 병렬 그룹 일부 반려 → 전체 rejected / S42 병렬 그룹 내 delegation 1건 skip → 나머지만으로 완료 / S43 병렬 그룹 내 대결 (proxy) 1건

### 1.4 Constraints

| Constraint | Details | Impact |
|---|---|---|
| **SvelteKit adapter-node 유지** | 서버리스 전환 없음 — puppeteer 로컬 프로세스 풀 운용 가능 | High (설계 전제) |
| **외부 SaaS 의존 금지** | timestamping authority / cloud PDF 서비스 등 제외. Docker 기반 self-hosted 만 허용 | Medium |
| **기존 RPC 시그니처 하위 호환** | v1.0/v1.1 호출부 코드를 수정 없이 동작시키기 어렵다면, 새 파라미터는 기본값으로 추가 (p_content_hash 는 `DEFAULT NULL` 후 CHECK 제약으로 강제) | High |
| **한국어 전용 PDF** | 한글 폰트 1종(나눔고딕) embed, 다국어 i18n 없음 | Low |
| **0001~0030 migration 수정 금지** | 누적만 허용. 0031~ 로만 확장 | High |

## 2. Alternatives Explored (Plan Plus Phase 2)

### 2.1 D1 — 병렬결재 데이터 모델

| Aspect | D1-A `group_order` column — **Selected** | D1-B Separate `approval_groups` table | D1-C JSONB `parallel_spec` |
|---|---|---|---|
| Summary | `approval_steps` 에 `group_order int NOT NULL`. 동일 값=병렬, 증가=순차. `current_step_index` 의미 재정의 | 신규 `approval_groups` 테이블 + `steps.group_id` FK | 문서에 JSONB 로 그룹 구조 저장 |
| Pros | 최소 schema 변경, 쿼리 단순, 기존 step_index 정렬 재활용 | 정규화 깔끔, 그룹 메타 부착 용이 (완료 조건 OR 등) | Schema 불변 |
| Cons | 필드명 drift (step_index vs group_order) — 문서화 필요 | 신규 테이블, RLS 정책 1건, 2-way join 증가, 복잡도 상승 | 쿼리/인덱스/감사 RLS 연동 모두 실패 |
| Effort | Low | Medium | N/A (탈락) |
| Best For | v1.2 AND-only | v1.3+ OR-조건·그룹 메타 필요 시 | 없음 |

**Decision**: **D1-A**. v1.2 YAGNI 를 따르면 그룹 메타가 불필요. D1-B 의 추가 복잡도는 가치 없음.

### 2.2 D2 — PDF 렌더 전략

| Aspect | D2-A' `puppeteer` (번들 Chromium) — **Selected** | D2-B Gotenberg Docker | D2-C `pdfkit` 프로그래매틱 |
|---|---|---|---|
| Summary | npm `puppeteer` (~150MB, 번들 Chromium 포함) + `/documents/[id]/pdf` 라우트 self-request `?print=1` | HTTP 기반 외부 Docker 서비스 | 의존성 경량 라이브러리로 PDF 프로그래매틱 생성 |
| Pros | 한글 OK (`@font-face` + Nanum embed), FormRenderer.svelte 재사용 → 렌더 로직 중복 제거, 화면 = PDF 픽셀 일치 | 메인 프로세스 메모리 격리, 장애 격리 | 의존성 소형, 메모리 미미 |
| Cons | Chromium 바이너리 크기, 메모리 150-300MB/req, 콜드스타트 2-3s | docker compose 추가 의존, self-hosted 고객 배포 부담 | form_schema 를 PDF 좌표로 재구현 필요 — **렌더 로직 2중 유지** (장기 부채) |
| Effort | Medium (프로세스 풀 1개 웜 유지) | Medium-High (compose 문서화 부담) | High (폰트 embed 수동, 레이아웃 재구현) |
| Best For | HTML 중심 동적 양식 | 초대규모 트래픽 + 전용 PDF 서비스 | 고정 템플릿만 쓰는 PDF |

**Decision**: **D2-A'**. 콜드스타트는 프로세스 풀(1개 싱글톤 + warm-up on boot)로 완화. "렌더 로직 1개" 라는 가치가 메모리 비용을 초과.

### 2.3 D3 — 해시 알고리즘·범위·위치·시점

| 축 | 옵션 | Selected | 근거 |
|---|---|---|---|
| Algorithm | SHA-256 / SHA-3/256 / BLAKE3 | **SHA-256** | 한국 공공·법무 호환성 최고, Node `crypto` 내장 |
| Canonicalization | RFC 8785 JCS / naive `JSON.stringify` | **RFC 8785 JCS** | 키 순서 독립성 — 해시 안정성 필수. 자체 구현 ~60줄 또는 `@truestamp/canonify` 채택 검토 |
| Scope | content / +schema/+line / +attachments | **+schema+line+attachments** | schema drift + 결재선 변조 + 첨부 교체 모두 감지. "감사" 관점에서 최소 |
| 저장 위치 | single column / history table | **single column** `approval_documents.content_hash` | 상신 후 불변(RPC 차단), 이력 불필요 |
| 계산 시점 | DB RPC / Node 서버 | **Node 서버** | JCS 구현이 JS에 훨씬 간결. pgcrypto 의존 최소화. RPC 는 strict equality 체크만 수행 (p_content_hash 인자 → 컬럼 저장) |

**Compensating control**: content_hash 를 audit_logs.payload 에도 mirror 저장 → Node 변조 시 DB 측 2중 기록으로 역추적 가능.

### 2.4 D4 — 서명이미지 저장

| Aspect | D4-A Profile 1-signature path — **Selected** | D4-B base64 직접 저장 | D4-C Document-ephemeral draw |
|---|---|---|---|
| Summary | `profiles.signature_storage_path text` + Storage `{tenant_id}/{user_id}.png` + 상신 snapshot to audit payload | profiles 에 base64 필드 | 매 승인 시 캔버스 드로잉 |
| Pros | 간단, 감사 스냅샷 자연, Storage 기존 버킷 패턴 활용 | 빠른 read | 진정한 서명 의미 |
| Cons | path 교체 시 구 서명 추적 필요 → audit payload 로 해결 | DB 사이즈 폭증 (10KB × users), RLS 복잡 | UX 복잡, 모바일 부담, YAGNI |
| Effort | Low | Medium | High |

**Decision**: **D4-A**. Compensating: 승인 시점의 `signature_storage_path + signature_sha256` 을 audit_logs.payload 에 snapshot.

### 2.5 Decision Rationale 종합

5개 축 모두 **"기존 v1.0/v1.1 로직 재사용"** 을 최상위 가치로 판단. 구체적으로:

- D1-A `group_order` — 순차 로직은 그룹 크기=1 의 특수 케이스로 자연 수용
- D2-A' puppeteer — FormRenderer.svelte 재사용 = 렌더 단일 source of truth
- D3 Node hash — 클라이언트·서버가 동일 JS 모듈 공유 (JCS 구현 1개)
- D4-A profile path — 기존 Storage 업로드 패턴 재사용
- G4 helper 추출 — v1.1 technical debt 를 v1.2 이익(병렬·순차 공용)으로 전환

모든 선택이 **"코드 path 가 1개"** 원칙을 지킴.

### 7.2 Key Decisions (한 표로 집약)

| Decision | Options | Selected | Rationale |
|---|---|---|---|
| 병렬결재 모델 | column vs table vs json | **`group_order` column** | 최소 schema 변경, 순차 특수케이스 자연 수용 |

| 해시 계산 위치 | DB vs Node | **Node** | JCS 구현 간결, 테스트 용이 |
| 해시 범위 | content-only vs full | **schema+content+line+attachments** | 감사 관점 완전성 |
| 서명 저장 | profile path vs base64 vs ephemeral | **profile path + audit snapshot** | 기존 Storage 패턴 재사용 |
| Helper 추출 | G4 분리 vs inline 유지 | **0031 으로 일원화** | 병렬·순차 공용 전제 |

### 7.4 Data Flow

#### (A) 병렬결재 submit

```
Client:
  1. 첨부 업로드 → per-file sha256 (crypto.subtle)
  2. approvalLine 작성 → "병렬 그룹 만들기" UI → items[].groupOrder 할당
     예: [A(0), B(0), C(1), D(2)]  — A와 B 병렬, 이후 C, 이후 D
  3. POST submit action

Server action:
  4. computeDocumentHash({ formSchema, content, approvalLine, attachmentSha256s })
     └─ JCS canonicalize → SHA-256 hex
  5. supabase.rpc('fn_submit_draft', {
       p_tenant_id, p_form_id, p_content, p_approval_line, p_content_hash, p_attachment_ids
     })

RPC (0035 fn_submit_draft v3):
  6. auth.uid() + is_tenant_member 검증
  7. approval_line 검증:
     - groupOrder 연속성 (set → sorted → 0,1,2,...N-1)
     - 각 group 에 ≥ 1개 'approval' step
     - 전체 approval step ≥ 1
  8. form_schema_snapshot SELECT FOR UPDATE
  9. approval_documents INSERT — content_hash = p_content_hash, current_step_index = 0
  10. approval_steps INSERT — 각 item 의 group_order 그대로, delegation 매칭 로직 (v1.1 M11 유지)
  11. fn_advance_document(v_doc.id) 호출 — 첫 pending group 찾기 → current_step_index 갱신 or 자동 completed
  12. attachments document_id 확정
  13. audit 'submit' 기록 — payload 에 content_hash mirror
  14. RETURN doc
```

#### (B) 병렬결재 approve (그룹 내 1건)

```
Client: /documents/[id] → Approve
Server action: supabase.rpc('fn_approve_step', { p_document_id, p_comment })

RPC (0036 fn_approve_step refactored):
  1. auth.uid() + document FOR UPDATE + 상태 체크 (v1.1 M13: 'in_progress' | 'pending_post_facto')
  2. 현재 step 잠금 (step_index = current_step_index) — **기존 로직 유지**
  3. proxy 분기 (v1.1 M12) 유지
  4. UPDATE step SET status='approved', acted_by_proxy_user_id = v_proxy
  5. fn_advance_document(v_doc.id) 호출 (교체):
     - current_group := approval_documents.current_step_index → 해당하는 group_order 파악
       (NEW 규칙: "current_step_index" 컬럼은 이제 group_order pointer 의미)
     - 동일 group_order 내에 여전히 pending approval step 있는지 확인
       └─ 있음 → doc 유지 (반환)
       └─ 없음 → next group_order 의 첫 pending approval step 탐색
            ├─ 있음 → current_step_index = next_group_order, 반환
            └─ 없음 → 남은 pending reference step 일괄 skipped + status=completed + completed_at
  6. audit 기록 (payload: step_index, signature snapshot, proxy 여부)
```

**중요**: `current_step_index` 컬럼의 의미를 **step_index 포인터 → group_order 포인터** 로 재정의. v1.0/v1.1 기존 문서는 group_order=step_index 로 backfill 되므로 값 동일 (정렬 시맨틱 우연적 일치). 이 재정의는 0032 migration 주석에 명시.

#### (C) PDF 렌더

```
Client: 완료 문서 페이지에서 "PDF 다운로드" 클릭
→ GET /t/{slug}/approval/documents/{id}/pdf

Route handler (+server.ts):
  1. locals.user / resolveTenant (일반 문서 라우트와 동일 권한 체크)
  2. const cookie = event.request.headers.get('cookie')
  3. renderDocumentPdf({ cookie, origin, docId })
  4. return new Response(pdfBuffer, {
       headers: {
         'Content-Type': 'application/pdf',
         'Content-Disposition': `attachment; filename="${docNumber}.pdf"`
       }
     })

renderDocument.ts (싱글톤):
  let browser: Browser | null = null
  async function getBrowser() {
    if (!browser) browser = await puppeteer.launch({ headless: 'new', args: ['--no-sandbox'] })
    return browser
  }
  async function renderDocumentPdf({ cookie, origin, docId }) {
    const page = await (await getBrowser()).newPage()
    try {
      // 세션 쿠키 전달
      await page.setExtraHTTPHeaders({ cookie })
      await page.goto(`${origin}/t/.../documents/${docId}?print=1`, { waitUntil: 'networkidle0' })
      return await page.pdf({ format: 'A4', printBackground: true, margin: {...} })
    } finally { await page.close() }
  }
```

`?print=1` 모드의 `documents/[id]/+page.svelte` 는 CSS `@media print` 대신 명시 prop 으로 헤더·탭·ApprovalPanel 을 hide — puppeteer 의 `print: true` 에뮬레이션보다 제어 용이.

### 9.1 `fn_submit_draft` (v3, 0035)

```sql
CREATE OR REPLACE FUNCTION public.fn_submit_draft(
  p_tenant_id      uuid,
  p_form_id        uuid,
  p_content        jsonb,
  p_approval_line  jsonb,          -- [{userId, stepType, groupOrder}]
  p_content_hash   text,           -- NEW: SHA-256 hex (64 chars)
  p_attachment_ids uuid[] DEFAULT '{}'::uuid[],
```

**v1.1 대비 변경점**:

- `p_approval_line` item 에 `groupOrder` 추가 (없으면 순차 — step_index 사용)
- 삽입 시 `group_order` 컬럼 채움
- 상신 끝에 `PERFORM fn_advance_document(v_doc.id)` — 0035 가 0031 helper 에 의존

**Validation 규칙 (신규)**:
- `p_content_hash` 필수, 길이 64, hex 패턴
- `approval_line` 내 groupOrder 집합이 `{0,1,...,K-1}` 연속 (set == range)

### 9.3 `fn_advance_document` (신규, 0031)

```sql
CREATE OR REPLACE FUNCTION public.fn_advance_document(p_document_id uuid)
RETURNS public.approval_documents
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_doc public.approval_documents;
  v_current_group int;
  v_next_group int;
  v_has_pending boolean;
BEGIN
  SELECT * INTO v_doc FROM public.approval_documents
  WHERE id = p_document_id FOR UPDATE;

  IF v_doc.id IS NULL THEN
    RAISE EXCEPTION 'Document not found' USING ERRCODE = '02000';
  END IF;

  -- 현재 group 에 pending approval 남아있는지
  v_current_group := v_doc.current_step_index;  -- 주: 의미 재정의됨 (group_order pointer)
  SELECT EXISTS (
    SELECT 1 FROM public.approval_steps
    WHERE document_id = p_document_id
      AND group_order = v_current_group
      AND step_type = 'approval'
      AND status = 'pending'
  ) INTO v_has_pending;

  IF v_has_pending THEN
    RETURN v_doc;  -- 아직 그룹 완료 전
  END IF;

  -- 다음 pending approval group 탐색
  SELECT MIN(group_order) INTO v_next_group
  FROM public.approval_steps
  WHERE document_id = p_document_id
    AND group_order > v_current_group
    AND step_type = 'approval'
    AND status = 'pending';

  IF v_next_group IS NULL THEN
    -- 모든 approval 완료 → document completed + 남은 reference 일괄 skipped
    UPDATE public.approval_steps
    SET status = 'skipped'
    WHERE document_id = p_document_id AND status = 'pending';

    UPDATE public.approval_documents
    SET status = 'completed',
        completed_at = now(),
        updated_at = now()
    WHERE id = p_document_id
    RETURNING * INTO v_doc;
  ELSE
    UPDATE public.approval_documents
    SET current_step_index = v_next_group,
        updated_at = now()
    WHERE id = p_document_id
    RETURNING * INTO v_doc;
  END IF;

  RETURN v_doc;
END;
$$;
```

### 9.4 `fn_approve_step` / `fn_reject_step` (refactor, 0036)

기존 0024/0026 의 "다음 step 탐색" 로직을 완전히 제거하고 `PERFORM fn_advance_document(v_doc.id)` 로 교체. Proxy 분기, audit 기록은 유지. audit payload 에 `signature_storage_path`, `signature_sha256` snapshot 추가.

| # | Risk | Impact | Likelihood | Mitigation |

## 14. Convention Prerequisites

### 14.1 RLS 작성 규칙 (v1.0/v1.1 유지)

- `auth.uid()` 는 반드시 `(SELECT auth.uid())`
- `FOR ALL` 금지, action 별 분리
- 같은 action 에 permissive 정책 중복 금지
- Helper 는 `SECURITY DEFINER + STABLE`
- 테이블 간 양방향 EXISTS 금지 — `SECURITY DEFINER` helper 로 우회

### 14.2 Mutation 규칙

- 모든 approval mutation 은 RPC 경유 (`ad_no_direct_insert` 유지)
- 상태 변경 시 `SELECT ... FOR UPDATE`
- 결재선·양식은 스냅샷 (content_hash 로 snapshot 불변 강제)

### 14.3 Postgres enum 규칙

- v1.2 에서는 새 enum value 추가 없음 (document_status, step_status, audit_action 모두 v1.1 그대로)
- 만약 추가 시 **TX 외부 파일 분리** (v1.1 M12 교훈 준수)

### 14.4 Svelte 5 규칙

- `$state(data.x)` 경고 → `untrack(() => [...data.x])`
- Hidden input 은 primitive 만, 복잡 state 는 `use:enhance({formData}) => formData.set(...)`
- `$lib/server/*` 를 `.svelte` 에서 import 금지 — client-safe 분리 (v1.1 M12 교훈)
- label for/id 또는 nesting (v1.1 교훈)

### 14.5 Storage 규칙

- 기존 `approval-attachments`: `{tenant_id}/{YYYY-MM}/{uuid}.{ext}` 유지
- 신규 `user-signatures`: `{user_id}/signature.png` (1인 1파일, upsert)
