# Plan — 그룹웨어 v3.1 확장

> Feature: `그룹웨어-v3.1-확장`
> Stack: SvelteKit 2 + Svelte 5 (runes) + @supabase/ssr + Supabase local
> Author: Rigel team
> Date: 2026-04-12
> Phase: PDCA Plan

---

## Executive Summary

| 항목 | 내용 |
|---|---|
| **Feature** | 그룹웨어 v3.1 — 게시판 + 공지사항 + 일정관리 + 근태관리 + 인사정보 |
| **타겟** | 기존 전자결재 사용 중인 SMB 테넌트 (10~300명) |
| **마일스톤** | M1~M2(P0) → M3(P1) → M4(P1) → M5(P2) |
| **Migration** | 0056~ (기존 0001~0055 유지) |
| **핵심 전환점** | "전자결재 SaaS" → "한국형 그룹웨어 SaaS" |

### Value Delivered (4관점)

| 관점 | 내용 |
|---|---|
| **Problem** | 전자결재만으로는 고객 Retention 한계. 게시판·공지·일정은 별도 도구(노션, 구글캘린더)에 분산 → 로그인 2~3곳, 조직 맥락 단절 |
| **Solution** | 기존 tenant/org/member 인프라 위에 게시판·공지·일정·근태·인사 5개 모듈 추가. 전자결재 연동(근태→결재, 공지→필독확인)으로 단일 플랫폼 효과 |
| **Function UX Effect** | 사이드바에 "소통(게시판/공지) · 업무(일정/근태) · 인사" 3섹션 추가. 기존 결재 흐름 유지하면서 원클릭 전환 |
| **Core Value** | "메일 + 게시판 + 전자결재 = 그룹웨어" (가비아 라이브러리 정의). v3.1 완료 시 경쟁 제품 최소 기능 동등 달성 |

## 1. 스코프 & 마일스톤

### 1.1 마일스톤 순서

| # | 기능 | 우선순위 | 의존성 | 예상 Migration |
|---|---|---|---|---|
| **M1** | 게시판 | P0 | 없음 | 0056~0058 |
| **M2** | 공지사항 | P0 | M1 (게시판 UI 패턴 재사용) | 0059~0060 |
| **M3** | 일정관리 | P1 | 없음 | 0061~0063 |
| **M4** | 근태관리 | P1 | 전자결재 연동 (기존 RPC) | 0064~0066 |
| **M5** | 인사정보 | P2 | 없음 | 0067~0068 |

### 1.2 구현 순서 근거

1. **M1 게시판이 최우선** — 한국 그룹웨어 3대 필수 (메일·게시판·결재) 중 게시판만 미구현
2. **M2 공지사항** — 게시판 테이블 구조 + UI 컴포넌트 위에 `is_pinned`, `require_read_confirm` 플래그 추가로 빠르게 구현
3. **M3 일정** — 독립 모듈, 게시판과 병렬 가능하나 UI 패턴 안정화 후 진행
4. **M4 근태** — 전자결재 연동이 핵심 (초과근무 신청 → approval_documents)
5. **M5 인사** — 기존 profiles 테이블 확장, 가장 단순

## 2. M1: 게시판 (Board)

### 2.1 기능 요구사항

| ID | 요구사항 | 설명 |
|---|---|---|
| B-01 | 게시판 생성/관리 | 관리자가 게시판 생성 (전사/부서별), 이름·설명·권한 설정 |
| B-02 | 게시글 CRUD | 제목, 본문(richtext — Tiptap 재사용), 파일첨부 |
| B-03 | 댓글 | 1-depth 댓글, 작성자/시간 표시, 삭제 |
| B-04 | 파일 첨부 | 기존 `approval-attachments` 버킷 패턴 재사용, 게시판용 별도 경로 |
| B-05 | 목록/검색/페이징 | 제목+본문 검색, 최신순, 페이징 (20건/page) |
| B-06 | 부서별 접근 제어 | RLS — 전사 게시판은 전 멤버, 부서 게시판은 해당 부서만 |
| B-07 | 조회수 | 게시글별 조회수 카운트 |
| B-08 | 고정글 | 관리자가 게시글 상단 고정 (is_pinned) |

### 2.2 DB 설계

```sql
-- 0056_boards.sql
CREATE TABLE boards (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id uuid NOT NULL REFERENCES tenants(id),
  name text NOT NULL,
  description text,
  board_type text NOT NULL DEFAULT 'general'
    CHECK (board_type IN ('general', 'department')),
  department_id uuid REFERENCES departments(id),
  created_by uuid NOT NULL REFERENCES auth.users(id),
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

-- 0057_board_posts.sql
CREATE TABLE board_posts (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  board_id uuid NOT NULL REFERENCES boards(id) ON DELETE CASCADE,
  tenant_id uuid NOT NULL REFERENCES tenants(id),
  title text NOT NULL,
  content jsonb NOT NULL DEFAULT '{}',  -- Tiptap JSON
  author_id uuid NOT NULL REFERENCES auth.users(id),
  is_pinned boolean NOT NULL DEFAULT false,
  view_count integer NOT NULL DEFAULT 0,
  file_paths text[],  -- Storage 경로 배열
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE board_comments (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  post_id uuid NOT NULL REFERENCES board_posts(id) ON DELETE CASCADE,
  tenant_id uuid NOT NULL REFERENCES tenants(id),
  author_id uuid NOT NULL REFERENCES auth.users(id),
  content text NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now()
);
```

### 2.3 RLS 전략

- `boards`: SELECT — `is_tenant_member(tenant_id)` + (general OR 본인 부서). INSERT/UPDATE/DELETE — admin/owner만
- `board_posts`: SELECT — 해당 board 접근 가능한 멤버. INSERT — 해당 board 멤버. UPDATE/DELETE — 작성자 OR admin
- `board_comments`: SELECT — post 접근 가능. INSERT — post 접근 가능. DELETE — 작성자 OR admin

### 2.4 라우트

```
/t/[slug]/boards/                    — 게시판 목록
/t/[slug]/boards/[boardId]/          — 게시글 목록
/t/[slug]/boards/[boardId]/new       — 게시글 작성
/t/[slug]/boards/[boardId]/[postId]  — 게시글 상세 + 댓글
/t/[slug]/admin/boards/              — 게시판 관리 (CRUD)
```

| C-06 | 캘린더 필터 | 개인/팀/전사/회의실 토글 |

### 4.2 DB 설계

```sql
-- 0061_calendar_events.sql
CREATE TABLE calendar_events (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id uuid NOT NULL REFERENCES tenants(id),
  title text NOT NULL,
  description text,
  start_at timestamptz NOT NULL,
  end_at timestamptz NOT NULL,
  all_day boolean NOT NULL DEFAULT false,
  event_type text NOT NULL DEFAULT 'personal'
    CHECK (event_type IN ('personal', 'department', 'company')),
  department_id uuid REFERENCES departments(id),
  created_by uuid NOT NULL REFERENCES auth.users(id),
  recurrence_rule text,              -- iCal RRULE (선택)
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  CHECK (end_at > start_at)
);

-- 0062_meeting_rooms.sql
CREATE TABLE meeting_rooms (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id uuid NOT NULL REFERENCES tenants(id),
  name text NOT NULL,
  location text,
  capacity integer,
  created_at timestamptz NOT NULL DEFAULT now()
);

-- 0063_room_reservations.sql
CREATE TABLE room_reservations (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  room_id uuid NOT NULL REFERENCES meeting_rooms(id) ON DELETE CASCADE,
  tenant_id uuid NOT NULL REFERENCES tenants(id),
  title text NOT NULL,
  reserved_by uuid NOT NULL REFERENCES auth.users(id),
  start_at timestamptz NOT NULL,
  end_at timestamptz NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now(),
  CHECK (end_at > start_at),
  -- 같은 회의실 시간 겹침 방지: EXCLUDE constraint
  EXCLUDE USING gist (
    room_id WITH =,
    tstzrange(start_at, end_at) WITH &&
  )
);
```

### 5.1 기능 요구사항

| ID | 요구사항 | 설명 |
|---|---|---|
| T-01 | 출퇴근 체크 | 버튼 클릭 → 현재 시각 기록 (GPS 없음, v3.1은 단순) |

| T-03 | 초과근무 신청 | 초과근무 시간 입력 → **전자결재 문서로 자동 생성** |
| T-04 | 관리자 현황 | 부서별 출근 현황, 지각/결근 통계 |
| T-05 | 근무 유형 | 정상/지각/반차/연차/출장/재택 — 기존 absence 연동 |

### 5.2 DB 설계

```sql
-- 0064_attendance_records.sql
CREATE TABLE attendance_records (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id uuid NOT NULL REFERENCES tenants(id),
  user_id uuid NOT NULL REFERENCES auth.users(id),
  work_date date NOT NULL,
  clock_in timestamptz,
  clock_out timestamptz,
  work_type text NOT NULL DEFAULT 'normal'
    CHECK (work_type IN ('normal', 'late', 'half_day', 'annual_leave', 'business_trip', 'remote')),
  note text,
  created_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE (tenant_id, user_id, work_date)
);

-- 0065_overtime_requests.sql (전자결재 연동)
-- 별도 테이블 없음 — 기존 approval_documents + form_code='overtime' 양식으로 처리
-- seed form: { code: 'overtime', fields: [date, hours, reason] }

-- 0066_attendance_settings.sql
CREATE TABLE attendance_settings (
  tenant_id uuid PRIMARY KEY REFERENCES tenants(id),
  work_start_time time NOT NULL DEFAULT '09:00',
  work_end_time time NOT NULL DEFAULT '18:00',
  late_threshold_minutes integer NOT NULL DEFAULT 10,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

### 5.3 전자결재 연동

- **초과근무 양식** (`overtime`): 기존 `form_schemas` 에 seed INSERT
- 기안 시 `approval_documents` 생성, 기존 결재 흐름 그대로 사용
- 승인 완료 시 → `attendance_records`에 초과근무 반영 (trigger 또는 RPC)

## 7. 공통 아키텍처

### 7.1 사이드바 네비게이션 확장

```
현재 사이드바:
  대시보드
  전자결재 (결재함, 기안, 문서)
  휴가 캘린더
  관리

v3.1 확장:
  대시보드
  전자결재 (결재함, 기안, 문서)
  ──────────
  소통
    게시판
    공지사항
  업무
    일정관리
    근태관리
    휴가 캘린더 (기존 이동)
  ──────────
  관리
```

### 7.2 Storage 전략

- 게시판/공지 첨부: `{tenant_id}/boards/{YYYY-MM}/{uuid}.{ext}` — 기존 `approval-attachments` 버킷 재사용 또는 `groupware-attachments` 신규 버킷
- 기존 업로드 유틸 (`$lib/client/uploadAttachment.ts`) 경로 파라미터만 변경하여 재사용

### 7.3 RLS 패턴 (기존 규칙 준수)

1. `(SELECT auth.uid())` InitPlan 래핑 — 모든 새 RLS 정책에 적용
2. `is_tenant_member(tenant_id)` 헬퍼 재사용
3. action별 개별 정책 (FOR ALL 금지)
4. 부서 접근 체크: `is_department_member(tenant_id, department_id)` 신규 헬퍼 필요

### 7.4 공통 컴포넌트 재사용

| 기존 컴포넌트 | 재사용 대상 |
|---|---|
| TiptapEditor | 게시글/공지 본문 |
| AdminNav | admin/boards, admin/announcements, admin/rooms, admin/attendance |
| OrgTreePicker | 부서 선택 (게시판 생성, 일정 부서 지정) |
| uploadAttachment | 게시판/공지 파일 첨부 |
