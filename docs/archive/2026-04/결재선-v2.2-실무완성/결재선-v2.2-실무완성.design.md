## 2. Data Model

### 2.1 Migration 0049 — approval_line_favorites

```sql
CREATE TABLE public.approval_line_favorites (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id uuid NOT NULL REFERENCES public.tenants(id) ON DELETE CASCADE,
  user_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  name text NOT NULL,
  form_id uuid REFERENCES public.approval_forms(id) ON DELETE SET NULL,
  line_json jsonb NOT NULL,  -- ApprovalLineItem[]
  created_at timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX alf_user_idx ON public.approval_line_favorites (tenant_id, user_id);
ALTER TABLE public.approval_line_favorites ENABLE ROW LEVEL SECURITY;

-- 본인만 CRUD
CREATE POLICY alf_owner_select ON public.approval_line_favorites
  FOR SELECT TO authenticated
  USING (user_id = (SELECT auth.uid()));

CREATE POLICY alf_owner_insert ON public.approval_line_favorites
  FOR INSERT TO authenticated
  WITH CHECK (user_id = (SELECT auth.uid()) AND public.is_tenant_member(tenant_id));

CREATE POLICY alf_owner_delete ON public.approval_line_favorites
  FOR DELETE TO authenticated
  USING (user_id = (SELECT auth.uid()));
```

```sql
ALTER TABLE public.approval_documents
  ADD COLUMN IF NOT EXISTS urgency text NOT NULL DEFAULT '일반';

COMMENT ON COLUMN public.approval_documents.urgency IS '일반 | 긴급';
```

### 2.4 Migration 0052 — fn_approve_step agreement 지원

```sql
-- fn_approve_step 내 agreement 분기 추가
-- disagreed → 문서 전체 rejected (approve와 동일 흐름이지만 action이 다름)
-- agreed → 다음 step 진행 (approve와 동일)
-- 기존 'approval'/'reference' 처리는 변경 없음
```

> 실제 SQL은 기존 fn_approve_step 전체를 CREATE OR REPLACE로 재정의.
> agreement step의 `disagreed` action은 reject와 동일하게 문서 status='rejected' 설정.

### 3.1 M1: 즐겨찾기 — `drafts/new/[formCode]/+page.server.ts` 확장

**load 추가**:
```typescript
// 현재 사용자의 즐겨찾기 (이 양식 또는 전체)
```

### 3.3 M3: 일괄 결재 — `inbox/+page.server.ts` 확장

**신규 actions**:

```typescript
batchApprove: async ({ request, locals, params }) => {
  const fd = await request.formData();
  const ids = JSON.parse(fd.get('documentIds')?.toString() ?? '[]') as string[];
  const comment = fd.get('comment')?.toString() ?? '';

  const results: { id: string; ok: boolean; error?: string }[] = [];
  for (const docId of ids) {
    const { error } = await locals.supabase.rpc('fn_approve_step', {
      p_document_id: docId,
      p_comment: comment
    });
    results.push({ id: docId, ok: !error, error: error?.message });
  }
  return { batchResult: results };
},

batchReject: async ({ request, locals, params }) => {
  // 동일 구조, fn_reject_step 호출
}
```

**변경 후**:
```
[양식명 + 설명]

[★ 즐겨찾기 | 📋 추천 | 🤖 자동]   ← M1+M2 통합
```

1. **결재선을 내용 위로** — 결재선을 먼저 확정하고 내용 작성 (실무 패턴)

3. **즐겨찾기/추천/자동 3단 구성** — 선택 우선순위 시각화
4. **StampLine 프리뷰** — 결재선 확정 시 가로 도장 미리보기

## 6. Risk & Mitigation

| # | 리스크 | 영향 | 대응 |
|---|---|---|---|
| R1 | 일괄 결재 partial failure | High | 개별 결과 배열 반환 + 실패 건 UI 표시 |
| R2 | agreement step 기존 문서 호환 | Medium | 기존 approval/reference 그대로 유지, agreement는 신규 문서만 |
| R3 | 즐겨찾기 line_json에 탈퇴 멤버 | Medium | 적용 시 userId → profiles 조회, 없으면 경고 표시 |
| R4 | 대시보드 6-count 쿼리 성능 | Low | GROUP BY status 단일 쿼리 |
| R5 | fn_approve_step 재정의 risk | High | 기존 테스트 케이스 유지 + agreement 분기만 추가 |
