## Lessons Learned

### What Went Well

- **DB trigger + pg_net 패턴 정착**: audit_logs INSERT → notifications 자동 생성 → Edge Function 비동기 호출 체인이 깔끔하게 작동. 기존 RPC 코드를 수정하지 않고 trigger만으로 additive 구현 완성
- **Edge Function console fallback 전략**: dev 환경에서 SMTP 없이도 이메일 내용 확인 가능 — 로컬 테스트 마찰 최소화
- **Realtime + SSR 이중 초기화**: layout.server.ts에서 unreadCount SSR 초기값 + layout.svelte Realtime 구독으로 hydration 깜빡임 없이 실시간 업데이트
- **1 iteration으로 79% → 91% 달성**: 핵심 gap(Edge Function 미구현, Apple PWA meta 누락)이 명확해 빠른 수정 가능

### Areas for Improvement

- **Analysis 초기 점수 79%**: Edge Function 전체 구현이 iteration 후로 밀린 것이 가장 큰 요인. 구현 계획 시 Edge Function을 M21에서 M19~M20으로 앞당기거나, M21 완료 확인 후 분석 실행 권장
- **반응형 커버리지**: Inbox 탭, ApprovalPanel 반응형이 M22에서 부분 적용 — 컴포넌트별 responsive 체크리스트를 Do 단계에서 명시적으로 관리하면 누락 방지 가능
- **manifest.json 품질 필드**: description, lang, shortcuts 등 PWA Lighthouse 점수 관련 필드는 Design에서 Must/Should로 구분해 기록하면 향후 gap 계산에서 혼선 감소

### To Apply Next Time

- **Edge Function은 별도 Milestone 마지막이 아닌 기능 완성 직후 통합**: 이메일 발송 채널은 trigger → pg_net → Edge Function 체인 전체가 검증되어야 M 완료로 간주
- **Apple PWA meta 체크리스트 Design에 명시**: `apple-mobile-web-app-capable`, `apple-touch-icon` 등은 표준 스펙 외 항목이므로 Design §10 PWA에 별도 항목으로 나열
- **반응형 적용 범위를 컴포넌트 단위로 체크박스화**: 레이아웃 컴포넌트 목록과 모바일 변경 내용을 Design에 표로 정리 (v1.3에서 일부 도입했으나 누락 발생)

## Key Technical Decisions

| 결정 | 선택 | 이유 |
|---|---|---|
| 이메일 채널 | Edge Function + SMTP (pg_net fire-and-forget) | TypeScript 템플릿, 로컬 테스트, Supabase 스택 완결 |
| Realtime 구독 대상 | notifications 전용 테이블 | RLS 단순화, 읽음 상태 관리 용이, audit_logs 직접 구독 회피 |
| 모바일 전략 | 반응형 CSS + PWA | 네이티브 앱/Capacitor 과도, SMB SaaS에 PWA 충분 |
| Realtime 채널 이름 | `notif:{tenantId}` (Design: `notif:{userId}`) | per-tenant 채널 + RLS filter로 동등 격리 (accepted gap G2) |
| 알림 데이터 정리 | pg_cron 90일 이상 read=true 자동 삭제 | R26 대용량 누적 방지 |
