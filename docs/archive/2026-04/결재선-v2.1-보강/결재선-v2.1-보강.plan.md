# Plan — 결재선 v2.1 보강 (결재선-v2.1-보강)

> Feature: `결재선-v2.1-보강`
> Stack: SvelteKit 2 + Svelte 5 (runes) + @supabase/ssr + Supabase local
> Date: 2026-04-12
> Phase: PDCA Plan
> 선행: v2.0 (match 78%) — DB/RPC 완성, UI MVP

## Executive Summary

| 항목 | 내용 |
|---|---|
| **Feature** | 결재선 자동 구성 규칙의 관리 UI 보강 + 기안 시 결재선 프리뷰 |
| **선행 버전** | v2.0 — approval_rules 테이블, fn_resolve_approval_line RPC, FormBuilder D&D |
| **Migration** | 0046~ (기존 0001~0045 수정 금지) |
| **핵심 목표** | 비개발자 관리자가 JSON 없이 결재선 규칙을 생성·관리할 수 있게 |
| **성공 기준** | 관리자가 규칙 빌더 GUI만으로 결재선 규칙 CRUD 완료 + 기안자가 제출 전 자동 결재선 확인 가능 |

### Value Delivered (4관점)

| 관점 | 내용 |
|---|---|
| **Problem** | v2.0에서 DB/RPC는 완성됐지만, 규칙 관리가 JSON textarea → 비개발자 관리자 사용 불가. 기안자는 자동 결재선이 누구인지 제출 전 확인 불가 |
| **Solution** | (1) 시각적 규칙 빌더 GUI, (2) 조직도 트리 + 직급 관리 UI, (3) 기안 시 자동 결재선 프리뷰, (4) 관리자용 규칙 시뮬레이터 |
| **Function UX Effect** | 관리자: 드롭다운·카드 기반 규칙 생성 → 시뮬레이션으로 즉시 검증. 기안자: 양식 작성 중 "결재선 자동 구성" 클릭 → 실제 결재자 이름·직급 즉시 확인 |
| **Core Value** | "IT 도움 없이 관리자가 직접 결재 정책을 설정하고, 기안자가 신뢰할 수 있는 결재선을 확인" = SMB 자립 운영 완성 |

## 2. Feature Details

### M1: ApprovalRuleBuilder GUI

**현재**: `/admin/approval-rules/+page.svelte` — JSON textarea로 `line_template_json` 직접 입력

**목표**: 시각적 빌더 컴포넌트

```
┌──────────────────────────────────────────────┐
│ 규칙: 일반 결재선                             │
│ ─────────────────────────────────────────── │
│ 적용 양식: [전체 ▼]  우선순위: [100]         │
│                                              │
│ ▼ 적용 조건 (선택)                           │
│  ┌─ 필드: [금액 ▼] 연산자: [>= ▼] 값: [___] │
│                                              │
│ ▼ 결재선 구성                                │
│  ┌─ Step 1: [팀장 ▼]    유형: [승인 ▼]  [×] │
│  ├─ Step 2: [부서장 ▼]  유형: [승인 ▼]  [×] │
│  └─ + 단계 추가                              │
│      ↳ 조건: 금액 >= 1000만원일 때만 포함    │
│                                              │
│ [저장]  [시뮬레이션]  [취소]                  │
└──────────────────────────────────────────────┘
```

**구현 포인트**:
- `RuleBuilder.svelte` 신규 컴포넌트
- role 드롭다운: `team_lead` | `department_head` | `ceo` | 직접 지정 (멤버 검색)
- stepType 드롭다운: `approval` | `agreement` | `reference`
- step별 조건(condition) 토글 — 접히는 패널
- JSON ↔ GUI 양방향 변환 (기존 JSON 규칙도 GUI로 편집 가능)

### M2: 기안 시 결재선 프리뷰

**현재**: `drafts/new/[formCode]/+page.svelte` — 수동 결재선만, 자동해석 결과 미표시

**목표**: "결재선 자동 구성" 버튼 → RPC 호출 → 결재자 프로필(이름·직급·부서) 표시

```
┌─────────────────────────────────────────────────┐
│ 결재선                                           │
│ ────────────────────────────────────────────── │
│ [결재선 자동 구성]  [초기화]                      │
│                                                  │
│  1. 김팀장 (영업팀/팀장)        승인    [×]       │
│  2. 박부장 (영업부/부장)        승인    [×]       │
│  3. 이대표 (경영지원/대표이사)  승인    [×]       │
│                                                  │
│  ℹ️ 규칙 "일반 결재선" 기반 자동 구성             │
│  └─ 자동 구성 후 수동 편집 가능                   │
└─────────────────────────────────────────────────┘
```

**구현 포인트**:
- 서버 action `resolveApprovalLine` 추가 — fn_resolve_approval_line RPC 호출
- 결과를 ApprovalLine 컴포넌트에 주입 (`line = resolved`)
- 프로필 조회: 2-query + Map join 패턴 (`$lib/server/queries/profiles.ts` 재사용)
- 자동 구성 후에도 수동 편집(추가/삭제/순서변경) 가능 유지

### M3: OrgChart 트리 UI

**현재**: `/admin/org/+page.svelte` — 평면 리스트, parent_id 미활용, "트리는 v1.1+"

**목표**: 트리 시각화 + parent 설정 + 부서 CRUD

```
┌─────────────────────────────────────────────┐
│ 조직도                                       │
│ ──────────────────────────────────────────── │
│ 📁 경영지원본부                               │
│   ├─ 📁 인사팀 (3명)                         │
│   ├─ 📁 재무팀 (2명)                         │
│   └─ [+ 하위 부서]                            │
│ 📁 영업본부                                   │
│   ├─ 📁 영업1팀 (5명)                        │
│   ├─ 📁 영업2팀 (4명)                        │
│   └─ [+ 하위 부서]                            │
│ [+ 최상위 부서 추가]                           │
└─────────────────────────────────────────────┘
```

**구현 포인트**:
- `OrgTree.svelte` 재귀 트리 컴포넌트 (DeptNode 반복)
- departments 테이블의 `parent_id` FK 활용 (0005에서 이미 존재)
- 인라인 이름 편집, 상위 부서 변경 (select), 삭제 (하위 있으면 경고)
- 부서별 멤버 수 badge (tenant_members count)

### M4: JobTitles 관리 UI

**현재**: job_titles 테이블 존재 (0041), tenant_members.job_title_id FK (0042). 관리 UI 없음

**목표**: `/admin/org` 페이지 내 직급 관리 탭/패널

- 직급 CRUD (name, level 1~10)
- 멤버별 직급 할당 (members 페이지 또는 org 페이지 내)
- level 의미 안내: 1=사원, 3=대리, 5=과장, 7=부장, 9=이사, 10=대표

### M5: 규칙 시뮬레이터

**목표**: 관리자가 규칙 저장 전/후에 "이 규칙으로 기안하면 누가 결재자인가" 테스트

- `/admin/approval-rules` 페이지 내 시뮬레이션 패널
- 입력: 양식 선택, 기안자(멤버 선택), 샘플 content (key-value)
- 출력: fn_resolve_approval_line 호출 결과 → 결재자 이름·직급 표시
- DB에 새 RPC 불필요 — 기존 fn_resolve_approval_line에 서버 action으로 호출

### M6: 조건부 필드 에디터 (P2)

**현재**: FormBuilder에서 conditionals를 편집하는 UI 없음 (FormRenderer에서 동작은 함)

**목표**: FormBuilder 내 조건 규칙 시각적 편집기

- "조건부 표시 규칙" 패널: if [필드 ▼] [== ▼] [값] then show/hide [필드들]
- 기존 ConditionalRule 타입 그대로 활용
- FormBuilder.svelte 하단에 ConditionalsEditor 섹션 추가

### M7: FormRenderer hidden 필터링 (P2)

**현재**: 조건부로 숨겨진 필드 값이 제출 content에 포함됨

**목표**: `drafts/new/[formCode]/+page.svelte`의 submit 시 hiddenFieldIds 기반 필터링

- FormRenderer가 이미 `hiddenFieldIds` derived를 계산함
- submit enhance에서 `content`에서 hidden field 키 삭제 후 전송
- 변경 최소: enhance 콜백 내 3~5줄 추가

## 3. DB/Migration 변경 사항

### 3.1 기존 테이블 변경 없음

v2.1은 **UI 보강 중심**. DB 스키마 변경 최소화.

| 대상 | 변경 | 이유 |
|---|---|---|
| approval_rules | 변경 없음 | condition_json, line_template_json 구조 그대로 활용 |
| departments | 변경 없음 | parent_id, path 이미 존재 (0005) |
| job_titles | 변경 없음 | 0041에서 완성 |
| tenant_members | 변경 없음 | department_id, job_title_id FK 이미 존재 |

### 3.2 가능한 Migration (0046~)

| # | 대상 | 내용 | 조건 |
|---|---|---|---|
| 0046 | departments | `parent_id` 에 인덱스 추가 (트리 쿼리 최적화) | parent_id 인덱스가 없을 때만 |
| 0046 | tenant_members | `department_id` 인덱스 확인 | 없으면 추가 |

> Design 단계에서 현행 인덱스 확인 후 확정. 새 테이블이나 RPC 추가는 예상하지 않음.

## 5. 구현 순서

```
Phase 1 (P0 — 핵심 UX)
  M4: JobTitles 관리 UI ─────────────┐
  M3: OrgChart 트리 UI ──────────────┤── 기반 데이터 관리
                                      ↓
  M1: ApprovalRuleBuilder GUI ────── 규칙 관리 GUI
                                      ↓
  M2: 기안 시 결재선 프리뷰 ──────── 기안자 경험 완성
                                      ↓
Phase 2 (P1 — 관리자 도구)
  M5: 규칙 시뮬레이터 ───────────── 관리자 검증 도구
                                      ↓
Phase 3 (P2 — 빌더 보완)
  M6: 조건부 필드 에디터 ─────────── 빌더 완성도
  M7: hidden 필드 필터링 ─────────── 데이터 정확성
```

**권장 구현 순서**: M4 → M3 → M1 → M2 → M5 → M6 → M7

이유:
1. M4(직급)·M3(조직도)가 먼저 — 규칙 빌더에서 role → 실제 사용자 매핑이 조직도·직급에 의존
2. M1(규칙 빌더)이 핵심 — 가장 복잡한 UI, 충분한 컨텍스트에서 작업
3. M2(프리뷰)는 M1 완성 후 — 같은 RPC 활용, 자연스러운 연장
4. M5(시뮬레이터)는 M1에 붙는 기능
5. M6·M7은 독립적, 마지막에

## 6. Risk & Mitigation

| # | 리스크 | 영향 | 대응 |
|---|---|---|---|
| R1 | RuleBuilder GUI 복잡도 — step 조건, 병렬 그룹, 멤버 검색 조합 | High | step별 조건은 접히는 패널로 점진 노출. 병렬은 v2.1에서 제외 (순차만) |
| R2 | fn_resolve_approval_line RPC 호출 시 부서/직급 미설정 멤버 | Medium | NULL 반환 시 "결재자를 찾을 수 없습니다" 안내 + 조직도 설정 링크 |
| R3 | 기존 JSON 규칙과 GUI 빌더 호환성 | Medium | JSON → GUI 파싱 실패 시 "고급 편집 (JSON)" 모드 fallback |
| R4 | OrgTree 깊은 재귀 (대규모 조직) | Low | SMB 타겟이므로 depth 5 이하. 필요시 lazy loading |
| R5 | departments parent_id 순환 참조 | Low | 서버 action에서 cycle 검사 (ancestor 체크) |

## 7. Non-Functional Requirements

| 항목 | 기준 |
|---|---|
| 접근성 | 모든 form control에 label 연결 (svelte-check a11y 0 warning) |
| 반응형 | OrgTree·RuleBuilder는 모바일에서 세로 스택 레이아웃 |
| 성능 | OrgTree 초기 로드 < 200ms (100 부서 기준) |
| 보안 | 모든 관리 action은 `is_tenant_admin` RLS 게이트 (기존 패턴 유지) |
| 하위호환 | 기존 JSON textarea로 생성된 규칙이 GUI에서 정상 로드·편집 가능 |

## 8. 검증 계획

| 항목 | 방법 |
|---|---|
| M1 규칙 빌더 | 관리자 계정으로 GUI에서 규칙 생성 → DB 확인 → 기존 JSON 규칙 GUI 로드 확인 |
| M2 결재선 프리뷰 | 기안 페이지에서 양식 작성 → "자동 구성" → 결재자 이름 표시 확인 |
| M3 조직도 | 부서 생성 → 하위 부서 추가 → parent 변경 → 삭제 (하위 있으면 경고) |
| M4 직급 | 직급 생성 → 멤버 할당 → fn_resolve_role에서 정상 매칭 확인 |
| M5 시뮬레이터 | 규칙 + 샘플 content → 결과 프리뷰 → 실제 기안 결과와 일치 확인 |
| M6 조건 에디터 | 조건 추가 → FormRenderer 프리뷰에서 표시/숨김 동작 확인 |
| M7 hidden 필터 | 숨겨진 필드에 값 입력 → 제출 → DB content에서 해당 키 없음 확인 |
| 통합 | svelte-check 0 error, npm run build 성공 |
