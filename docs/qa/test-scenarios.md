# Rigel v3.1 테스트 시나리오 (전체)

> 전체 기능별 수동 테스트 체크리스트. 프로덕션 배포 전 회귀 검증, 신규 설치 후 기본 동작 확인용.

## 테스트 환경 기준

| 항목 | 값 |
|---|---|
| 대상 | `https://app.rigelworks.io` (또는 개발 `http://localhost:5173`) |
| 테스트 계정 | seed.sql 3개 + 샘플 데이터 옵션 on 시 자동 생성 |
| 데이터 상태 | seed + 샘플 (부서 3, 양식 5, 게시판 2, 공지 2, 회의실 3) |

### 계정 정보

| 이메일 | 비밀번호 | 역할 | 부서 | 직급 |
|---|---|---|---|---|
| `m3test1@example.com` | `testpass1234` | owner | 경영지원팀 | 부장 |
| `m5approver@example.com` | `testpass1234` | member | 개발팀 | 과장 |
| `newuser@example.com` | `testpass1234` | member | 영업팀 | 사원 |

### 브라우저 준비

- DevTools Console 열어두기 → `[rigel:api-fail]`, `[rigel:unhandled]` 태그로 실패 즉시 가시화
- `?debug=1` 쿼리 또는 `localStorage.rigel_debug='true'` 로 verbose 모드
- 시크릿 창 별도 탭으로 다른 사용자 동시 로그인 권장

### 표기법

- ✅ PASS 조건 / ❌ FAIL 조건 / ⚠️ 주의
- **[O]** = owner, **[A]** = admin, **[M]** = member, **[G]** = 비로그인 게스트

## 1. 인증 & 온보딩

### 1.1 신규 가입 (로컬 개발) — `/signup`

1. `/signup` 접속 → 이름/이메일/비밀번호(8자+) 입력 → 가입
2. ✅ 자동 로그인 → `/onboarding/create-tenant` 이동
3. ❌ 비밀번호 7자 이하 → 클라이언트 validation
4. ❌ 이미 사용 중 이메일 → 서버 에러 메시지 표시

### 1.2 로그인 / 로그아웃

1. `/login` — 등록된 계정 로그인 → `/` → tenant 있으면 `/t/{slug}` 리다이렉트
2. 잘못된 비밀번호 → "이메일 또는 비밀번호가 올바르지 않습니다"
3. 로그인 후 `/login` GET → `/` 리다이렉트 (이중 로그인 방지)
4. 헤더 우측 **로그아웃** 클릭 → 세션 제거 → `/login`

### 1.3 조직 생성 (온보딩) — `/onboarding/create-tenant`

1. 조직명 입력 → slug 자동 추천 (영소문자/숫자/하이픈)
2. **[✓] 샘플 데이터 함께 생성** 체크 (기본 on)
3. 생성 → `/t/{slug}/approval/inbox` 이동
4. ✅ 샘플 체크 시 대시보드에 부서·게시판·공지·회의실 채워져 있음
5. ❌ slug 2자 이하/32자 초과 → validation
6. ❌ 이미 사용 중 slug → "이미 사용 중인 주소입니다"

### 1.4 초대 수락 — `/invite/{token}`

1. **[O]** `관리 > 멤버` 에서 초대 발급 → 토큰 URL 복사
2. 다른 브라우저에서 URL 접속 → 가입 화면
3. 가입 완료 → 자동으로 해당 tenant에 member로 추가
4. ❌ 만료/사용된 토큰 → 안내 메시지

## 2. 전자결재 — Core

### 2.1 품의서 기안 → 상신

**역할**: M (newuser)

1. 사이드바 `+ 새 기안` → `품의서` 선택
2. 결재자 선택 → **m3test1(부장)** 추가 (조직 드롭다운 → 경영지원팀)
3. 문서 설정: 긴급도=긴급, 보관기한=5년, 보안등급=일반
4. 내용 입력: 제목/목적/예산/타당성
5. 첨부파일 (선택): PDF 1개 업로드
6. **상신** 클릭 → `/approval/inbox` 로 이동
7. ✅ 내 문서 탭에 새 문서 보임 (상태: 상신진행)

### 2.2 승인

**역할**: O (m3test1) — 상기 기안의 결재자

1. `/approval/inbox` → 결재대기 탭
2. 신규 품의서 카드 클릭 → `/approval/documents/{id}`
3. 본문·결재선·첨부 확인
4. **승인** 버튼 → 코멘트(선택) → 확인
5. ✅ 토스트 또는 페이지 리로드, 상태 "완료"로 변경
6. ✅ 기안자(newuser)에게 **알림 생성** (배지 +1, 드롭다운에 "[결재 완료]")

### 2.3 반려

1. 기안자가 새 품의서 상신
2. 결재자가 **반려** + 코멘트 입력 (예: "예산 재검토 필요")
3. ✅ 문서 상태 "반려"로 변경
4. ✅ 기안자에게 반려 알림 + 코멘트 표시

### 2.4 회수

1. 기안자가 상신 후 결재자가 아직 처리 안 한 상태
2. 문서 상세 → **회수** 버튼 → 확인
3. ✅ 상태 "회수"로 변경, 결재 단계 모두 취소
4. ❌ 결재자가 한 명이라도 처리했으면 회수 불가 (버튼 숨김)

### 2.5 재기안

1. 반려된 문서 상세 → **재기안**
2. 기존 내용이 템플릿으로 복사된 새 draft
3. 수정 후 상신 → 새 문서로 생성
4. ✅ 이전 반려 문서와 링크 표시

### 2.6 댓글

1. 문서 상세 하단 코멘트 영역
2. 결재자·기안자·참조자가 댓글 추가 가능
3. ✅ 실시간으로 다른 사용자 화면에 댓글 반영 (realtime)

## 3. 전자결재 — 고급

### 3.1 전결 (Delegation)

**선행**: **[O]** `관리 > 전결 규칙` 에서 규칙 등록
- 위임자: m3test1, 대리자: m5approver, 양식: 품의서, 한도: 100만원

1. newuser가 품의서 기안 (예산 50만원), 결재자: m3test1
2. 상신 → `fn_submit_draft`가 delegation 매칭
3. ✅ m3test1 step이 `skipped` + comment `[전결] <m5approver id>`
4. ✅ 실제 결재는 m5approver가 수행
5. ✅ audit_logs에 `delegated` 액션 기록
6. 변형: 한도 초과 (150만원) → delegation 무시, 원래대로 m3test1 결재

### 3.2 대결 (Proxy Approval)

1. m3test1이 부재 중 (`관리 > 부재 관리`에 등록 + proxy: m5approver)
2. newuser가 품의서 상신, 결재자: m3test1
3. ✅ m3test1의 결재 단계가 proxy(m5approver)로 자동 전환
4. ✅ audit_logs에 `approved_by_proxy` 기록

### 3.3 후결 (Post-facto)

1. 기안 화면 → **[✓] 후결 (사후 결재)** 체크 + 사유 입력 필수
2. 상신 → 상태 `pending_post_facto`
3. 결재자 승인 시 일반 승인과 동일
4. ✅ 타임라인에 "후결 사유: ..." 표시

### 3.4 병렬 결재 (Parallel)

1. 기안 시 결재선 구성: 같은 groupOrder 에 2명 배치
2. 상신 → ✅ 해당 group의 두 결재자 모두 pending
3. 두 명 다 승인해야 다음 group 진행
4. 한 명이 반려 → 즉시 문서 전체 반려

### 3.5 합의 (Agreement)

1. 결재선에 **stepType: agreement** 추가 (결재 아닌 합의 의견)
2. 합의자가 `동의` / `비동의` 선택
3. ✅ 동의: 다음 단계 진행
4. ✅ 비동의: 문서 전체 반려 + audit `disagreed`

### 3.6 일괄 결재 / 일괄 반려

1. Inbox 결재대기 탭 → 체크박스로 2건 이상 선택
2. **일괄 승인** 또는 **일괄 반려** 버튼
3. ✅ 선택 문서 모두 처리, 결과 토스트
4. ❌ 본인 결재 단계 아닌 문서 선택 시 skip

### 3.7 독촉

1. 기안자가 자기 문서 상세 → **독촉** 버튼
2. ✅ 현재 결재자에게 알림 발송
3. 하루 1회 제한

## 4. 양식 관리

### 4.1 양식 빌더 — `/admin/forms/builder/{id}`

1. **[O/A]** `관리 > 양식 > 품의서` → **빌더**
2. 좌측 필드 팔레트에서 드래그 or 클릭으로 추가 (텍스트/장문/리치/숫자/날짜/기간/선택/첨부)

6. ❌ 빈 양식 저장 시 "최소 1개 필드 필요"
7. ✅ 새 양식 즉시 `/approval/drafts/new/{code}` 에서 사용 가능

### 4.2 양식 버전 관리

1. 이미 상신된 문서가 있는 양식 수정 → **발행** 시 새 버전 INSERT (parent_form_id 링크)
2. ✅ 이전 버전으로 상신된 문서는 `form_schema_snapshot` 에 독립 저장 → 영향 없음
3. ✅ 새 기안은 최신 버전으로만 가능

### 4.3 시스템 양식 복사

1. 신규 조직 생성 시 `fn_copy_system_forms` 자동 실행
2. ✅ 5개 양식 (품의서·업무협조전·지출결의서·휴가신청서·일반 기안서) 모두 복사됨
3. ✅ 각각 tenant-local 레코드, 수정 가능

## 7. 첨부파일

### 7.1 업로드

1. 기안 화면 "근거 자료" 첨부 → 파일 선택 (10MB 이하)
2. ✅ 진행률 표시 → 업로드 완료 → sha256 해시 계산
3. 파일 경로: `{tenant_id}/{YYYY-MM}/{upload_uuid}.{ext}` (Storage)
4. ❌ 10MB 초과 / 바이러스·exe 등 → validation

### 7.2 다운로드

1. 문서 상세 → 첨부파일 아이콘 클릭
2. ✅ 원본 파일명으로 다운로드 (Content-Disposition 복원)
3. ✅ signed URL TTL 만료 후 재접근 시 링크 재발급

### 7.3 해시 검증

1. 상신된 문서의 `content_hash` 표시 (관리자 감사 UI)
2. ✅ 본문 또는 첨부 sha256이 상신 시점 해시와 일치 확인 가능

## 10. 공지사항

### 15.3 읽음 처리

2. ✅ 배지 카운트 즉시 -1 ($derived 재계산)
3. **모두 읽음** 버튼 → 전부 read=true + 배지 0

## 18. 멀티테넌시 / RLS

### 18.1 테넌트 격리

1. A 조직 owner로 로그인

3. ✅ 404 또는 403 (멤버가 아니면 접근 불가)

### 18.2 RLS 정책

1. newuser(member)가 다른 사용자 문서 URL 직접 접근
2. ✅ 본인 기안/참조자/결재자가 아니면 404
3. 관리자는 조직 내 모든 문서 접근 가능 (0072)

### 18.3 Storage 격리

1. A 테넌트의 첨부 URL을 B 테넌트 멤버가 접근 시도
2. ✅ signed URL TTL 만료되면 403
